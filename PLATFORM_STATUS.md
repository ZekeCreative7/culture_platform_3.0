# Culture Platform 3.0 — Status & Scale-Up Planning Reference

> 작성일: 2026-06-17
> 목적: 현재까지 구현된 기능, 아키텍처, 알려진 한계를 정리하여 다음 단계(스케일업) 계획 수립의 기준 자료로 사용.

---

## 1. 한 줄 요약

Lina Life Insurance 조직문화 세션(팀빌딩/팀장/크로스펑셔널)을 운영하기 위한 **단일 운영자용 브라우저 SPA**. 인증 없음, 백엔드는 Firestore 단일 컬렉션 구조, 데이터는 `localStorage`-우선 + Firestore 동기화 하이브리드. 38개 커밋을 거쳐 Streamlit 프로토타입 → 바닐라 JS 웹앱으로 전환 완료.

## 2. 현재 기술 스택

| 영역 | 현재 상태 |
|---|---|
| 프론트엔드 | Vanilla JS (ES Modules), 빌드 시스템 없음, 4,000+ 줄 단일 `app.js` |
| 스타일 | 순수 CSS (`styles.css`, 1,100+ 줄), CSS Grid + Flexbox, 디자인 토큰(CSS 변수) |
| 백엔드 | Firebase Firestore v10.14.1 (CDN import, SDK 미설치) |
| 인증 | **없음** — 누구나 URL 접근 시 전체 읽기/쓰기 가능 |
| 데이터 저장 | `localStorage` (`culture-platform-webapp-v1`) 우선 기록 + Firestore 비동기 동기화 |
| 호스팅 | GitHub Pages (`https://zekecreative7.github.io/culture_platform_3.0/webapp/`) |
| QR 생성 | 클라이언트 사이드 (`qrcode.min.js`), 실패 시 외부 API 폴백 |
| 레거시 | Streamlit 프로토타입(`app.py`, `db.py`, `schema.sql`) 보존 — 참조용, 더 이상 개발 안 함 |

### Firestore 컬렉션 구조 (현재)

```
sessions/{id}          — 세션 (1기, 2기... 기수 + 유형 + 일정)
surveys/{id}           — 배포된 설문 메타데이터 + 동적 문항 (q1~q11 가변)
responses/{id}         — CSV 업로드된 익명 응답 (사전/중간/사후)
appState/main          — 전체 상태 스냅샷 단일 문서 (sessions/surveys/orgUnits/orgMembers/qualAnalysis 통째로)
```

**구조적 한계**: `appState/main`은 모든 데이터를 하나의 문서에 담는 단일 글로벌 블롭. 멀티 테넌시, 동시 편집, 부분 업데이트가 불가능. 현재는 수동 "DB 전송 / DB 다운로드" 버튼으로만 동기화되는 보조 경로이며 사실상 백업/복원 용도로 쓰임.

## 3. 기능별 구현 현황

### 3.1 Sessions (세션 관리)
- 세션 유형 3종: 팀빌딩(8주) / 팀장(4주) / 크로스펑셔널(6주), 각기 다른 회차 템플릿
- 회차별 일정(날짜/시간/소요시간/내용/메모) 편집, 확정 여부 체크
- 세션 생성/수정/삭제 → Firestore 자동 동기화 (`setDoc`/`deleteDoc`)
- 조직도에서 팀 선택 시 부문장/본부장/팀장/팀원 인원 자동 표시
- 크로스펑셔널 세션: 팀장 세션에서 인원 차출 또는 랜덤 샘플링 모드

### 3.2 Organization (조직도)
- 전사 → 부문 → 본부 → 팀 4단계 계층 구조 (정적 `org_data.json`, 273KB)
- 조직원 검색, 직책 별칭 매핑(CEO→사장, 전무→부사장 등)
- 조직도 피커 UI로 세션 생성 시 팀 선택

### 3.3 Survey Creator (설문 설계 및 배포)
- 동적 문항 설계 (5점 척도 / 주관식), 문항 ID 자동 부여
- 구글 폼 URL 연결 방식(권장) 또는 자체 설문 직접 설계 방식 병행
- 기존 설문에서 문항 템플릿 불러오기
- 로컬 QR 코드 생성 + PNG 다운로드, 링크 복사
- 설문 카드 접기/펼치기 (개별 + 전체 일괄), 2열 자동 그리드 문항 에디터
- CSV 결과 수동 업로드 경로 (구글 폼 응답을 사후에 적재할 때 사용)

### 3.4 Upload (CSV 업로드)
- `[기수]`, `[q1]~[q8]`, 선택적 `[q9]~[q11]` 컬럼 검증
- 이름/이메일/사번 등 개인식별자 컬럼 저장 차단
- 업로드 전 5행 프리뷰 + 검증 오류 표시

### 3.5 Change Analysis (변화 분석)
- 기수 + 세션 유형 선택 → 사전/중간/사후 비교
- **Pulse Overview**: 종합 점수(4개 영역 평균) 큰 숫자 카드 + RAG 색상 + delta
- 4대 영역(심리적 안전감/사일로 해소/회복탄력성/전반 분위기) 요약 스트립
- 문항별 상세 바 차트 + 통계 테이블
- 정성 응답 섹션 (AI 분석 연결)

### 3.6 Report (전문가 분석 리포트)
- **① 현 상황 진단**: SVG 레이더 차트(4축) + 영역별 RAG 점수 카드 + 종합 진단 요약 문장
- **② 세션 운영 제안**: 영역별 점수 구간에 따른 사전 작성된 퍼실리테이션 가이드 (심리적 안전감 — Amy Edmondson 프레임 기반 멘트 포함)
- **③ 변화 분석**: 영역별 사전→중간→사후 비주얼 바 카드, N<3 마스킹
- **④ 정성 응답 AI 분석**: GPT/Claude에 붙여넣을 프롬프트 자동 생성 → 결과를 `## 섹션` 형식으로 받아 파싱 → 키워드 pill, 인용구 blockquote 등 구조화된 카드로 렌더링 (구조 미감지 시 평문 폴백)
- CSV 다운로드

### 3.7 전역 UI/UX
- 접을 수 있는 사이드바 (데스크탑, 64px 아이콘 모드)
- DB 연결 상태 표시 점 (connecting/connected/error)
- 모바일 슬라이드 드로어 내비게이션 + 백드롭
- 한국어/영어 병기 메뉴 레이블
- 비비드 블루 디자인 시스템 (그라디언트, 글로우, 카드 섀도우)

## 5. 알려진 한계 / 기술 부채

### 5.1 보안 (스케일업 시 최우선)
- **인증 없음** — Firestore 보안 규칙이 열려 있다면 누구나 전체 조직 데이터를 읽고 쓸 수 있음
- 단일 운영자 가정 — 역할 구분(관리자/퍼실리테이터/조회 전용) 없음
- `appState/main` 단일 문서에 조직원 PII 인접 데이터(직책/이름)가 포함되어 있을 가능성 — 접근 통제 없이 노출 위험

### 5.2 데이터 아키텍처
- `localStorage`가 1차 소스, Firestore는 사실상 백업/동기화 — 두 기기에서 동시 작업 시 마지막 쓰기가 이전 기록을 덮어씀 (CRDT/충돌 해결 없음)
- `appState/main` 통짜 문서 구조는 문서 크기 제한(1MiB)에 조직 규모가 커지면 도달 가능
- 설문 응답(`responses`)에 사용자 식별자가 없어 한 사람의 사전/중간/사후 응답을 종단 추적 불가 (의도된 익명성이지만, 코호트 단위 집계만 가능하고 개인 변화 추적 불가능)
- 정성 분석 결과(`qualAnalysis`)가 AI에 의해 수동 복사/붙여넣기로 채워짐 — API 연동 없음, 자동화 안 됨

### 5.3 코드 구조
- `app.js` 단일 파일 4,000+ 줄 — 모듈 분리 없음, 빌드 시스템 없음 (현재는 의도적 선택이었음)
- 빌드/번들링 도구 없음 → 코드 분할, 트리 셰이킹, 타입 체크 불가
- 테스트 코드 없음 (단위/통합/E2E 전무)
- 인라인 스타일이 상당수 (`style="..."` 직접 작성) — 디자인 시스템과 별개 경로로 스타일 누적

### 5.4 운영
- 정적 `org_data.json` (273KB) — 조직 변경 시 파일 재배포 필요, 실시간 반영 안 됨
- CSV 업로드가 유일한 응답 수집 경로 (구글 폼 결과를 수동으로 다운로드 → 업로드) — 자동 동기화 없음
- 환경별 분기(`localhost` 감지 등)가 하드코딩 — 스테이징 환경 없음

## 6. 스케일업 시 검토할 방향 (참고, 결정 아님)

아래는 현재 한계를 바탕으로 도출한 **검토 후보**이며, 우선순위/범위는 별도 계획에서 확정 필요.

1. **인증 + 역할 기반 접근 제어** — Firebase Auth 또는 별도 IdP, 관리자/퍼실리테이터/조회자 역할 분리
2. **데이터 모델 정규화** — `appState/main` 통짜 문서를 컬렉션별 정규화된 문서로 전환, 부분 업데이트 가능하게
3. **멀티 테넌시 고려** — 여러 고객사/조직이 동일 플랫폼을 쓸 경우 조직 단위 데이터 격리 구조 필요 (현재는 Lina 단일 조직 전용으로 설계됨)
4. **구글 폼 응답 자동 수집** — Google Sheets API 또는 Apps Script 웹훅으로 CSV 수동 업로드 단계 제거
5. **AI 분석 자동화** — 현재 "프롬프트 복사 → ChatGPT 붙여넣기 → 결과 복사 → 저장"의 4단계 수동 루프를 OpenAI/Anthropic API 직접 호출로 전환 가능
6. **코드 구조화** — 빌드 도구(Vite 등) 도입 시 모듈 분리, 타입 안정성(TS 또는 JSDoc) 확보 가능 — 단, "빌드 없음" 현재 선택은 의도적이었으므로 트레이드오프 재논의 필요
7. **개인 추적 vs 익명성 트레이드오프** — 현재 가드레일(N<3 마스킹, 개인 식별자 미저장)을 유지하면서 코호트 종단 분석을 강화할 방법 검토

## 7. 참고 — 가드레일 (변경 시 주의)

`CLAUDE.md`에 명시된 불변 원칙, 스케일업 계획 시에도 유지 필요:
- 개인 수준 매칭/개인 식별자 추가 금지
- 경영진 리포트의 N<3 마스킹 유지
- Pulse Survey는 세션 변화 공식과 별개 컨텍스트로 취급 (사용자가 명시적으로 범위 변경하지 않는 한)
