## 1. 한 문장 목표

Pulse Survey를 조직과 직원을 평가하거나 곧바로 개입 대상을 정하는 도구가 아니라,

> **어디의 어떤 이야기를 먼저 듣고, 회사가 무엇을 인정하고 응답해야 하는지 찾는 기초체력 진단 도구**

로 재설계한다.

최종 운영 흐름은 아래와 같다.

```text
Pulse 기초체력 진단
→ 먼저 들어야 할 조직·주제 발견
→ 경청·공감 세션 연결
→ 경영진 응답과 작은 약속 등록
→ You said → We did 확인
→ 신뢰가 회복된 뒤 공동설계·행동 변화·90일 정착으로 이동
```

## 2. 이번 개선의 핵심 판단

### 2.1 현재 데이터가 말하는 핵심 스토리

현재 Pulse 분석에서 가장 먼저 전달해야 할 내용은 평균 점수나 부문 순위가 아니다.

- 웰빙: 2025년 42% → 2026년 59%로 개선
- 설문 조치 신뢰 Q19: 36% → 37%로 정체
- Q19 적극 부정: 26%
- 포용 Q20: 57% → 47%, 적극 부정 15% → 25%
- 소속 Q21: 63% → 56%
- 펀더멘털 Q1·Q2·Q4: 70% → 62%

따라서 첫 화면의 해석은 다음이어야 한다.

> 회사가 웰빙에 투자하고 있다는 신호는 전달되었지만, 직원의 목소리가 실제 변화로 이어진다는 믿음과 소속감은 회복되지 않았다. 지금은 변화 행동을 요구하기보다 “회사가 듣고, 인정하고, 작은 것이라도 바꾼다”는 경험을 만드는 것이 먼저다.

### 2.2 제품 포지션

- Pulse Survey = 연 1회 전사 기초체력 진단
- 세션 설문 = 세션 전·중·후 변화 확인
- 두 산식은 합치지 않는다.
- Pulse는 세션 효과 점수가 아니라 **경청·지원 방향을 정하는 컨텍스트**다.
- 낮은 점수는 벌점이나 조직장 평가가 아니다.
- 부문 순위는 운영자 내부 작업 큐로만 사용한다.

## 5. 이전 화면 정보 구조 초안 — 구현 기준으로 사용하지 않음

> 아래 3탭 구조는 기능을 구분하는 데는 유용하지만, 사용자가 스스로 분석의 이야기를 조립해야 한다는 문제가 있다. 실제 구현은 뒤의 **§5A 스토리텔링 화면 구조**를 최종 기준으로 사용한다. 아래 내용은 세부 기능 요구사항 참고용으로만 남긴다.

기존 `전사 개요 / 우선순위` 2개 탭을 아래 3개 탭으로 재구성한다.

### 탭 A. 전사 기초체력

첫 진입 화면이다. 순위보다 회사 전체의 감정과 신뢰 상태를 먼저 보여준다.

#### A-1. 신뢰 회복 헤드라인

표시 내용:

- 제목: `회사는 지원하고 있지만, “말하면 바뀐다”는 믿음은 아직 회복되지 않았습니다.`
- 부제: `지금은 변화를 요구하기보다 먼저 듣고, 인정하고, 작은 약속을 지킬 단계입니다.`
- 연도 선택
- 쉬운 말 / 전문 보기

데이터가 다르면 고정 문장을 쓰지 말고 `trustRecoveryHeadline(...)`에서 조건별 문장을 생성한다.

#### A-2. 지금 가장 중요한 4개 신호

큰 점수 카드가 아니라 해석형 신호 카드로 표시한다.

1. `설문 조치 신뢰` — Q19 FAV / UNFAV / 전년 대비
2. `안전하게 말할 수 있음` — Q17 FAV / UNFAV
3. `포용과 소속` — Q20·Q21 Net 및 전년 대비
4. `회사가 신경 쓴다는 신호` — Q11·Q12 웰빙 평균 및 전년 대비

카드에는 숫자 1개, 방향, 짧은 해석만 둔다. 상세 분포는 펼침 영역에 둔다.

#### A-3. Voice → Impact Gap

직원이 말할 수 있는 정도와 실제 조치가 일어났다고 믿는 정도의 차이를 보여준다.

```js
voiceCapacity = mean(FAV(Q5), FAV(Q17), FAV(Q18))
actionTrust = FAV(Q19)
voiceImpactGap = voiceCapacity - actionTrust
```

표시 원칙:

- `voiceImpactGap`은 공식 지수가 아니라 진단용 보조 신호라고 명시한다.
- 차이가 크면 `말할 수는 있지만, 말한 뒤 무엇이 달라졌는지 보이지 않습니다.`
- Q5·Q17·Q18 중 누락이 있으면 존재하는 값만 평균한다.
- Q19가 없으면 패널 전체를 `데이터 없음`으로 처리한다.

#### A-4. 좋아진 것과 약해진 것

현재 `테마 발산 지도`와 `좋아진 것 vs 약해진 것`을 유지하되 스토리 순서를 바꾼다.

- 잘된 것부터 인정: 웰빙, 매니저 지원 등
- 이어서 약해진 것: 포용, 소속, 추천, 자부심, 잔류
- FAV 하락이 중립으로 갔는지 UNFAV로 갔는지 구분
- `식는 중`과 `적극적으로 돌아섬`을 다른 언어로 설명

#### A-5. 우리가 먼저 들어야 할 세 가지

자동 생성되는 전사 아하 인사이트 3개를 표시한다.

각 카드 구조:

```text
제목
데이터 근거
해석 가설
세션에서 확인할 질문
경영진이 피해야 할 반응
```

가설을 사실처럼 단정하지 않는다.

#### A-6. 데이터 신뢰 경계

- 응답자 N
- N<3 마스킹
- 이상치 조직
- 조직개편 영향
- 동일 산식 확인 여부
- 인과가 아니라 가설 생성용이라는 문구

이 박스는 모든 분석 화면에서 접근 가능해야 한다.

#### B-5. 경청·지원 순서

- `시급` → `먼저 지원`
- `주의` → `대화 필요`
- `양호` → `강점 확인`
- `약점` → `먼저 들을 주제`
- `위험 본부` → `지원이 먼저 필요한 본부`

기본 화면에는 1등부터 전체 순위를 노출하지 않는다.

- 기본: 상위 5개 지원 후보 카드
- 전문 보기: 전체 작업 큐 테이블
- 경영진 공유용 출력에는 순위 숫자를 숨긴다.

### 탭 C. 신뢰 회복

분석이 실제 응답으로 닫히는 화면이다.

#### C-1. You said → We heard → We will → We did

각 약속 카드에 다음을 표시한다.

```text
You said     직원에게서 들은 핵심 주제
We heard     회사가 이해하고 인정한 내용
We will      실제로 할 수 있는 작은 약속
Owner        담당 역할 또는 부서
Due          완료 예정일
Status       검토 중 / 공유됨 / 진행 중 / 완료 / 보류
Evidence     완료 증거 또는 변경 내용
Source       연결된 연도·본부·문항
```

운영 원칙:

- 전사 또는 본부별 동시 활성 약속은 최대 2개를 권장한다.
- 할 수 없는 것은 솔직하게 `지금은 하기 어려움`으로 기록할 수 있어야 한다.
- 완료 근거 없는 `완료` 처리를 막는다.
- 약속은 개인 이름보다 담당 역할/조직을 기본으로 한다.

#### C-2. 신뢰 회복 상태

최초 버전에서는 복잡한 점수 대신 다음 카운트만 보여준다.

- 들은 주제 수
- 경영진 응답 대기
- 공유한 약속
- 진행 중 약속
- 완료 증거가 있는 약속

Q19 변화는 연간 Pulse에서만 확인하고, 약속 완료율과 섞어 하나의 지수로 만들지 않는다.

#### C-3. 세션 연결

각 인사이트 또는 약속 카드에서 현재 세션 생성 화면으로 이동할 수 있게 한다.

전달할 컨텍스트:

```js
{
  source: "pulse",
  pulseYear,
  scopeId,
  scopeLabel,
  questionIds,
  insightTitle,
  listeningQuestion,
  recommendedTone: "listen" | "acknowledge" | "co-design" | "practice"
}
```

초기에는 `sessionStorage` 또는 현재 앱 state에 draft context로 전달한다. 세션 데이터 구조에 Pulse 점수를 합산하지 않는다.

#### C-4. 경영진 응답문 프롬프트

복사 가능한 GPT 프롬프트를 제공한다.

출력 요청 형식:

1. 우리가 들은 것
2. 직원 감정에 대한 인정
3. 방어하거나 해명하지 않는 설명
4. 바로 할 수 있는 작은 약속 1~2개
5. 지금 하기 어려운 것과 이유
6. 다음 확인 일정
7. 피해야 할 표현

프롬프트는 수치 근거와 해석 가설을 명확히 구분해야 한다.

## 5A. 최종 구현 기준 — 스토리텔링 화면 구조

### 5A.1 구조 원칙

첫 화면은 여러 분석 기능을 모아놓은 대시보드가 아니라, 아래 질문에 순서대로 답하는 **하나의 세로 이야기**여야 한다.

```text
1. 무슨 일이 달라졌나?
2. 직원들은 지금 어떤 상태인가?
3. 왜 좋은 활동이 신뢰로 이어지지 않았나?
4. 어디의 어떤 이야기를 먼저 들어야 하나?
5. 회사는 무엇을 인정하고 답할 것인가?
6. 다음에 무엇을 확인할 것인가?
```

최상위 탐색은 기능명이 아니라 독자의 목적에 맞춘다.

- `한눈에 보기` — 전사 스토리를 처음부터 끝까지 읽는 기본 화면
- `조직별로 보기` — 본부를 선택해 같은 스토리 구조로 확대
- `근거 자세히` — 문항, 분포, 산식, 이상치, 전체 작업 큐

`신뢰 회복`은 별도 탭에 고립시키지 않는다. `한눈에 보기`와 `조직별로 보기`의 마지막 장에서 항상 회사의 응답과 작은 약속으로 연결한다.

### 5A.2 정보 카테고리 재구성

기존 카테고리는 분석가에게는 익숙하지만 일반 사용자는 서로의 관계를 한 번에 이해하기 어렵다. 쉬운 말 화면에서는 22개 문항을 **결과 1개 + 경험 동인 5개**로 재분류한다.

#### 결과 신호

| 사람이 이해하는 질문 | 문항 | 의미 |
|---|---|---|
| `직원들의 마음이 회사에 붙어 있는가?` | Q1~Q4 | 추천, 자부심, 성취감, 잔류 의향 |

#### 경험 동인

| 사람이 이해하는 질문 | 문항 | 화면 이름 |
|---|---|---|
| `일을 이해하고 성장할 수 있는가?` | Q6~Q10 | 일의 명확성과 성장 |
| `지치지 않고 도움받을 수 있는가?` | Q11~Q12 | 에너지와 돌봄 |
| `리더가 일상에서 돕고 인정하는가?` | Q13~Q16 | 리더와의 일상 |
| `안전하게 말하고, 말한 것이 바뀌는가?` | Q5, Q17~Q19 | 목소리와 실행 신뢰 |
| `여기에 속해 함께 일하고 있는가?` | Q20~Q22 | 소속과 연결 |

구현 원칙:

- 모든 22개 문항은 한 번만 기본 카테고리에 포함한다.
- `결과 신호`와 `경험 동인`을 같은 레벨의 점수 카드로 섞지 않는다.
- 전문 보기에서는 기존 `DOMAINS`, `THEMES`, `MANAGER_CLUSTER`, `CORE`를 유지한다.
- 동인이 결과와 함께 움직이는 것은 가설이며 인과로 단정하지 않는다.

### 5A.3 `한눈에 보기` — 일곱 장면

#### 장면 0. 이 화면을 보는 이유

> 이 결과는 누가 문제인지 찾기 위한 평가표가 아니라, 지금 무엇을 먼저 듣고 응답해야 하는지 정하기 위한 출발점입니다.

연도, 기준일, 쉬운 말/전문 보기는 오른쪽 상단 보조 영역에 둔다. 업로드는 `데이터 관리` 접힘 메뉴로 내린다.

#### 장면 1. 올해의 한 문장

화면에서 가장 큰 영역이다.

- 제목 예시: `회사의 돌봄은 전달됐지만, “말하면 바뀐다”는 믿음과 소속감은 약해졌습니다.`
- 설명: `웰빙은 좋아졌습니다. 하지만 설문 이후 실제 변화에 대한 믿음은 제자리이고, 포용과 소속은 낮아졌습니다.`
- 방향: `지금은 행동을 요구하기보다 먼저 듣고 작은 약속을 지킬 단계입니다.`

문장은 `trustRecoveryHeadline(...)`에서 실제 데이터 조건에 따라 생성한다. 특정 연도의 결론을 고정 문구로 박아두지 않는다.

#### 장면 2. 세 가지 변화

4~6개의 KPI 카드 대신 이야기 카드 3개만 보여준다.

1. `분명히 좋아진 것` — 에너지와 돌봄
2. `아직 바뀌지 않은 것` — 설문 조치 신뢰
3. `오히려 약해진 것` — 포용과 소속

각 카드에는 `사람이 이해하는 제목 / 대표 숫자 1개 / 이전→현재 / 의미 1문장`만 둔다.

`전사 핵심지표`, `본부 평균`, `분석 대상 수` 같은 운영 숫자는 첫 화면에서 제거하고 `근거 자세히`로 이동한다.

#### 장면 3. 숫자 사이에서 발견한 이야기

전문 용어 `깨진 커플링` 대신 서로 다르게 움직인 신호를 쉬운 문장으로 설명한다.

우선 표시할 이야기 예시:

1. `프로그램은 좋아졌지만, 말한 것이 바뀐다는 믿음은 낮습니다.`
2. `매니저의 지원은 나아졌지만, 회사 전체에 대한 소속감은 약해졌습니다.`
3. `남아 있으려는 마음보다 추천·자부심이 더 빠르게 식고 있습니다.`

기본 표시:

- 무엇과 무엇이 다르게 움직였는지
- 이것이 의미할 수 있는 가설
- 확인해야 할 질문 1개

근거 문항, gap, FAV/UNFAV는 `근거 보기` 펼침 안에 둔다.

#### 장면 4. 가장 먼저 회복해야 할 것은 신뢰

Voice → Impact Gap을 이 장면의 주인공으로 둔다.

```js
voiceCapacity = mean(FAV(Q5), FAV(Q17), FAV(Q18))
actionTrust = FAV(Q19)
voiceImpactGap = voiceCapacity - actionTrust
```

- 제목: `말할 수 있는 정도와, 말한 것이 바뀐다고 믿는 정도`
- 해석: `직원들은 어느 정도 말할 수 있지만, 그 뒤 무엇이 달라졌는지는 충분히 보지 못하고 있습니다.`
- 주의: `공식 평가지수가 아닌 대화 방향을 찾기 위한 보조 신호입니다.`

Q19가 없으면 계산을 숨기고 `설문 이후 조치 신뢰 문항이 없어 확인할 수 없습니다.`를 표시한다.

#### 장면 5. 어디의 이야기를 먼저 들을까

전사 순위표를 바로 보여주지 않는다.

- 지원 후보 본부 최대 5개
- 왜 먼저 들어야 하는지 한 문장
- 먼저 들을 주제 1개
- 경청 세션에서 물을 질문 1개
- `조직별로 보기` 이동 버튼

순위 숫자, priority 점수, 전체 표는 `근거 자세히`에만 둔다.

#### 장면 6. 그래서 회사가 할 일

분석이 직원 행동 요구로 끝나지 않고 회사의 응답으로 끝나야 한다.

```text
You said   우리가 들은 것
We heard   회사가 인정하고 이해한 것
We will    이번에 지킬 수 있는 작은 약속
We did     완료 후 보여줄 증거
```

- 활성 약속은 전사 기준 1~2개만 강조한다.
- 약속이 없으면 `새 프로그램을 시작하기 전에, 먼저 답해야 할 주제를 정하세요.`를 표시한다.
- `경영진 응답문 만들기`와 `경청 세션 연결`을 이 장면에 둔다.
- 30일/90일 성과는 다음 단계이며 첫 화면에서 요구하지 않는다.

#### 장면 7. 숫자를 믿을 수 있는 범위

이야기의 맨 끝에서 N, N<3 마스킹, 이상치, 조직개편, 동일 산식 여부, 인과 해석 제한을 짧게 설명한다. 긴 기술 설명은 `근거 자세히`로 이동한다.

### 5A.4 `조직별로 보기` — 같은 이야기의 확대

본부가 바뀌어도 화면 순서를 바꾸지 않는다.

```text
이 본부의 한 문장
→ 전사와 다른 점 3개
→ 먼저 들어야 할 경험 영역
→ 숫자 사이 이야기
→ 경청 질문
→ 회사/리더 응답
```

- 본부 선택과 함께 표본/이상치/조직개편 배지를 표시한다.
- `낮음/위험`으로 시작하지 않고 현재 경험을 한 문장으로 설명한다.
- 심리학적 분석과 조직 운영 분석은 가장 중요한 해석 1개씩만 기본 노출한다.
- 나머지 전문 해석은 펼침 처리한다.
- 본부별 결론 뒤에는 반드시 `경청 질문`과 `응답 방향`이 붙는다.

### 5A.5 `근거 자세히` — 전문 분석과 검증

다음 정보는 여기로 모은다.

- 2024~2026 전 문항 추세
- FAV / Neutral / UNFAV / Net
- 기존 도메인·테마별 계산
- 본부 전체 경청·지원 작업 큐
- priority 구성 요소
- 이상치와 조직개편 판정 근거
- 응답 N과 누락값
- GPT 상세 분석 프롬프트
- 데이터 업로드와 템플릿

전문 정보는 삭제하지 않고 **읽는 순서에서 뒤로 보낸다.**

### 5A.6 스토리 이해도 기준

첫 화면은 다음 테스트를 통과해야 한다.

- 30초 안에 `좋아진 것 / 문제인 것 / 지금 해야 할 것`을 말할 수 있다.
- Q번호나 FAV·UNFAV를 몰라도 결론을 이해할 수 있다.
- 스크롤 순서만 따라가도 별도 설명 없이 이야기가 이어진다.
- 각 장면은 하나의 질문에만 답한다.
- 한 화면 구간에 핵심 숫자는 최대 3개만 강조한다.
- 모든 문제 제기 뒤에는 `확인 질문` 또는 `회사 응답`이 따라온다.

## 6. 쉬운 말 / 전문 보기 원칙

정보는 한 번에 모두 펼치지 않고 세 단계로 공개한다.

```text
1단계 결론: 사람이 이해하는 한 문장과 대표 숫자
2단계 이해: 왜 그런지 보여주는 관계와 확인 질문
3단계 근거: 문항, 분포, 산식, 표본, 전체 표
```

쉬운 말과 전문 보기는 같은 정보를 단순히 번역한 화면이 아니다. 쉬운 말은 **의사결정의 순서**, 전문 보기는 **검증의 근거**를 제공한다.

### 쉬운 말

- 결론과 감정 상태를 먼저 보여준다.
- 숫자는 카드당 1개를 원칙으로 한다.
- 순위, 가중치, 상관계수는 숨긴다.
- `평가`가 아니라 `경청과 지원` 언어를 사용한다.

### 전문 보기

- FAV / UNFAV / Net
- 문항 분포
- 전년 대비 pp
- priority 구성 요소
- 이상치 판정 근거
- 조직개편 플래그
- 문항 간 gap
- N과 데이터 누락

### 용어 교체표

| 사용 자제 | 기본 노출 표현 |
|---|---|
| 변화 수용도 | 변화할 수 있는 조직 여건 |
| 위험 본부 | 지원이 먼저 필요한 본부 |
| 본부별 순위 | 경청·지원 순서 |
| 취약 문항 | 먼저 들어야 할 주제 |
| 개입 우선순위 | 대화 우선순위 |
| 변화 민첩성 | 신뢰·회복·변화 준비 상태 |
| 불만 | 부정 응답 / 적극 부정 응답 |
| 처방 | 확인 질문 / 지원 방향 |

전문 보기 또는 내부 코드에서는 기존 용어를 유지해도 되지만 쉬운 말 화면에는 노출하지 않는다.

## 8. 데이터 저장 계약

### 8.1 기존 Pulse 결과

기존 구조를 그대로 유지한다.

```text
pulseResults/{year}
```

분석용 파생값은 저장하지 않고 렌더 시 계산한다. 산식 변경 시 과거 저장값이 낡는 문제를 피하기 위함이다.

### 8.2 신규 신뢰 회복 약속

별도 컬렉션을 권장한다.

```text
pulseCommitments/{commitmentId}
```

권장 스키마:

```js
{
  id,
  year,
  scopeType: "company" | "division",
  scopeId,
  sourceQuestionIds: [19, 20, 21],
  employeeVoice: "",
  acknowledgement: "",
  commitment: "",
  ownerRole: "",
  dueDate: "YYYY-MM-DD",
  status: "draft" | "shared" | "in_progress" | "done" | "deferred",
  evidence: "",
  createdAt,
  updatedAt,
  completedAt: null
}
```

검증 규칙:

- `done`은 evidence가 없으면 저장 거부
- 개인 식별정보 입력 경고
- N<3 데이터에서 나온 직접 인용은 저장하지 않음
- 전사/본부 단위만 허용
- 삭제보다 `deferred` 또는 보관 상태를 우선

현재 인증이 없으므로 이 컬렉션을 운영 환경에 공개하기 전에 Firestore 접근 통제를 반드시 검토한다. 인증이 준비되지 않았다면 1차 구현에서는 로컬 상태로만 저장하고 UI에 `이 기기 저장`을 명시한다.

### 8.3 state 추가

```js
pulseView: "overview" | "listening" | "trust"
pulseScopeId: "company" | divisionId
pulseCommitments: []
pulseCommitmentDraft: null
pulseExpertSections: {}
```

기존 `pulseDeptId`는 `pulseScopeId`로 점진적으로 대체하거나 호환 처리한다.

## 8A. 연도별 데이터 추가와 자동 재분석

### 8A.1 결론

새 연도 데이터가 들어올 때 분석과 스토리가 자동으로 바뀌어야 한다. 2026년 결론을 화면에 고정해서는 안 된다.

현재 코드에는 다음 고정값이 있어 먼저 개선해야 한다.

- `PULSE_YEARS = [2024, 2025, 2026, currentYear + 1]`
- `DEFAULT_YEARS = [2024, 2025, 2026, currentYear + 1]`
- `itemMovements(yearDocs, 2025, 2026)`
- 화면 문구 `2025 → 2026`
- `pulseYear: 2026`
- `companyEngagement(...)`의 2026년 예외 처리
- 업로드의 22개 문항 고정 검증과 2025 조직개편 baseline

이 상태에서는 2027 데이터가 저장되어도 일부 화면이 계속 2025→2026을 설명하거나, 이후 연도 목록에서 누락될 수 있다.

### 8A.2 연도 자동 탐색

Firestore의 `pulseResults` 컬렉션에서 실제 존재하는 문서 ID를 읽어 연도 목록을 만든다.

```js
availableYears = Object.keys(pulseCache.years)
  .map(Number)
  .filter(Number.isFinite)
  .sort((a, b) => a - b)
```

초기 로드에서는 `getDocs(collection(db, "pulseResults"))`로 연도 카탈로그를 가져온다.

선택 원칙:

- 기본 선택 = 데이터가 존재하는 가장 최신 연도
- 사용자가 선택한 연도가 존재하면 선택 유지
- 삭제되거나 없는 연도면 가장 최신 연도로 폴백
- `currentYear + 1`은 템플릿 다운로드 목표 연도에만 사용

### 8A.3 비교 연도 자동 선택

선택 연도의 바로 이전 숫자가 아니라, **실제 데이터가 있고 비교 가능한 가장 가까운 과거 연도**를 선택한다.

```js
comparisonPair(yearDocs, selectedYear)
// { currentYear, previousYear, commonQuestionIds, coverage }
```

예시:

- 2027과 2026이 모두 있으면 2026→2027
- 2027은 있고 2026이 없으면 2025→2027, 화면에 `2026 데이터 없음` 표시
- 공통 문항이 부족하면 추세 결론을 만들지 않고 비교 불가 안내

화면의 연도 문구는 모두 `previousYear → currentYear`에서 생성한다.

### 8A.4 스토리 자동 생성

새 데이터가 들어오면 아래 순서로 스토리를 다시 계산한다.

```text
1. 현재와 직전 비교 가능 연도 결정
2. 결과 신호와 경험 동인별 변화 계산
3. 가장 좋아진 동인, 정체된 동인, 가장 약해진 동인 선택
4. Q19·Q17·Q20·Q21 등 신뢰 회복 핵심 문항 상태 확인
5. 문항 간 불일치 재계산
6. 본부 경청·지원 순서 재계산
7. 새 헤드라인과 확인 질문 생성
```

`trustRecoveryHeadline(...)`은 최소한 다음 경우를 구분한다.

- 돌봄은 상승, 신뢰·소속은 하락
- 신뢰는 회복, 소속은 정체
- 전반 상승이지만 특정 적극 부정 증가
- 전반 하락
- 비교 연도 없음
- 데이터 신뢰도 부족

2026년 문장은 검증용 예시일 뿐, 결과로 하드코딩하지 않는다.

### 8A.5 문항이 동일한 경우와 바뀌는 경우

#### 설문 22문항이 유지되는 경우

- 새 연도 업로드 즉시 기존 분석 전체를 재계산한다.
- 별도 코드 수정 없이 추세, 스토리, 본부 분석이 바뀌어야 한다.

#### 문항 문구·번호·척도가 바뀌는 경우

번호만으로 과거와 비교하면 안 된다. Pulse 문서 메타데이터에 다음을 저장한다.

```js
meta: {
  year,
  schemaVersion: "pulse-22-v1",
  formulaVersion: "pulse-story-v1",
  scale: "likert-5",
  questionCatalog: [
    { id: "Q19", semanticKey: "survey_action_trust", categoryKey: "voice_impact" }
  ],
  comparableWith: [2026],
  uploadedAt
}
```

원칙:

- 같은 `semanticKey`와 같은 척도만 추세 비교
- 신규 문항은 현재값만 표시하고 `신규 문항` 배지
- 삭제 문항은 과거 기록만 유지
- 문구가 바뀌어 의미가 달라졌으면 새 `semanticKey` 사용
- 비교 가능 여부가 불명확하면 자동으로 이어 붙이지 않음

초기 구현은 22문항 유지 시나리오를 지원하되, `schemaVersion`과 `semanticKey`를 지금부터 저장해 향후 변경을 막지 않는다.

### 8A.6 과거 기록 보존

- 새 연도 업로드는 과거 `pulseResults/{year}`를 덮어쓰지 않는다.
- 약속과 경영진 응답은 생성 당시 `year`, `scopeId`, `sourceQuestionIds`에 묶는다.
- 새로운 분석이 과거 약속의 문구나 완료 상태를 자동 변경하지 않는다.
- 산식이 바뀌면 `formulaVersion`을 올리고 화면에 버전을 표시한다.

### 8A.7 필수 검증

테스트용 2027 데이터를 만들어 다음을 확인한다.

- 2027이 자동으로 기본 선택됨
- 모든 `2025→2026` 문구가 `2026→2027`로 변경됨
- Q19가 크게 개선되면 `신뢰가 정체`라는 기존 헤드라인이 사라짐
- 웰빙이 하락하면 `돌봄이 좋아졌다`는 문장이 나오지 않음
- 본부 데이터 변화에 따라 지원 후보와 경청 질문이 달라짐
- 문항 하나가 누락돼도 공통 문항으로 비교하고 coverage 표시
- schemaVersion이 다르면 비교 중단 또는 경고

## 9. UI 디자인 원칙

- 현재 Culture Platform 3.0 디자인 시스템을 유지한다.
- 화면 제목은 `전사 추이`, `도메인 기준선` 같은 보고서 명사보다 `돌봄은 좋아졌지만 신뢰는 따라오지 않았습니다`처럼 결론형 문장을 우선한다.
- 각 섹션 상단에는 `이 장면이 답하는 질문`이 명확해야 한다.
- 첫 화면에서 여러 카드가 같은 중요도로 경쟁하지 않게 한다. 한 장면당 주인공 시각요소는 하나만 둔다.
- 차트는 숫자를 반복 장식하는 용도가 아니라 전환점, 격차, 발산 중 하나를 설명할 때만 사용한다.
- 같은 데이터를 카드·차트·표로 세 번 반복하지 않는다.
- 첫 화면을 빨강·주황 상태 카드로 채우지 않는다.
- 빨강은 적극 부정 급증과 데이터 경고에만 제한한다.
- 신뢰 회복 영역은 차분한 청색·청록·중립 회색을 사용한다.
- 카드 수를 늘리기보다 섹션별 핵심 메시지를 명확히 한다.
- 직원 발언은 장식용 인용구가 아니라 근거와 맥락을 함께 표시한다.
- 데스크톱은 2열, 모바일은 1열로 자연스럽게 접는다.
- 표는 모바일에서 가로 스크롤을 허용하되 핵심 요약은 표 밖에 둔다.
- 업로드 패널은 기본 접힘 상태로 바꿔 분석 흐름을 방해하지 않게 한다.
- 장면과 장면 사이에는 다음 내용을 예고하는 짧은 연결 문장을 둔다.
- 모바일에서도 이야기 순서를 바꾸지 않는다. 2열 카드를 1열로 쌓되 DOM 순서를 그대로 유지한다.

## 10. 구현 단계와 우선순위

### Phase 0 — P0: 연도 자동화 기반

목표: 새 연도 데이터를 업로드하면 코드 수정 없이 선택 연도, 비교 연도, 분석, 헤드라인, 본부 지원 순서가 다시 계산되게 한다.

작업:

1. Firestore `pulseResults` 실제 문서에서 연도 카탈로그 로드
2. 최신 데이터 연도를 기본 선택
3. `comparisonPair(...)`로 직전 비교 가능 연도 자동 선택
4. `2025`, `2026` 고정 비교와 화면 문구 제거
5. 헤드라인을 현재 데이터 기반 조건문으로 생성
6. `schemaVersion`, `formulaVersion`, `semanticKey`, 비교 가능 메타데이터 추가
7. 2027 가상 데이터로 스토리 변경 검증

완료 기준:

- 새 연도 문서를 추가한 뒤 코드 배포 없이 새 연도가 화면에 나타난다.
- 가장 최신 연도가 자동 선택되고 실제 직전 데이터와 비교된다.
- 새 데이터 방향이 달라지면 헤드라인과 이야기 카드도 달라진다.
- 비교 불가능한 문항을 같은 문항처럼 이어 붙이지 않는다.
- 과거 연도 데이터와 당시 약속은 변경되지 않는다.

### Phase 1 — P0: 신뢰 회복 중심 전사 개요

목표: 데이터 구조 변경 없이 첫 화면을 `무슨 일이 달라졌나 → 왜 그런가 → 지금 무엇을 할까`가 이어지는 하나의 이야기로 바꾼다.

작업:

1. 최상위 탐색을 `한눈에 보기 / 조직별로 보기 / 근거 자세히`로 변경
2. 결과 1개 + 경험 동인 5개의 질문형 카테고리 추가
3. `올해의 한 문장` 스토리 헤더 추가
4. `좋아진 것 / 바뀌지 않은 것 / 약해진 것` 3개 이야기 카드 추가
5. Q19, Q17, Q20·21, Q11·12 신호를 스토리 안에 배치
6. 문항 간 불일치 이야기 3개 추가
7. Voice → Impact Gap 추가
8. 지원 후보 본부와 경청 질문 연결
9. 마지막 장면에 회사 응답과 작은 약속 연결
10. 업로드와 전문 표는 `근거 자세히`로 이동
11. 용어 교체와 데이터 신뢰 경계 적용

완료 기준:

- 첫 화면에서 본부 순위보다 설문 신뢰와 소속 문제가 먼저 보인다.
- 첫 화면을 30초 본 사용자가 `좋아진 것 / 문제인 것 / 지금 해야 할 것`을 설명할 수 있다.
- Q번호, FAV, UNFAV를 몰라도 핵심 결론을 이해할 수 있다.
- 스크롤 순서가 `변화 → 의미 → 원인 가설 → 경청 대상 → 회사 응답`으로 이어진다.
- 첫 화면의 이야기 카드 수는 3개, 강조 숫자는 한 장면당 최대 3개다.
- 쉬운 말 화면에 `위험`, `개입`, `약점`, `수용도`가 기본 문구로 노출되지 않는다.
- 2026 데이터에서 Q19 37%, UNFAV 26%, 웰빙 59%, 포용·소속 하락이 정확히 표시된다.

## 15. 코드 검증 결과 및 보완 지시 (리뷰 첨부 · 2026-06-20)

### 15.1 검증 결론

### 15.3 반드시 반영할 보완 항목

계획 본문에 빠져 있거나 과소평가된 지점이다. 해당 Phase 착수 전에 반영한다.

1. **localStorage 영속화 화이트리스트 (Phase 0/1/3 공통, 필수)**
   `saveState()`는 저장 키를 명시적으로 화이트리스트한다(`state.js:164`~`187`). §8.3에서 추가하는 상태(`pulseScopeId`, `pulseCommitments`, `pulseCommitmentDraft`, `pulseExpertSections` 등)는 `blankState()`(`state.js:34`~)와 `saveState()` **양쪽 모두**에 추가해야 한다. 한쪽만 추가하면 새로고침 시 값이 사라진다. §12.2의 "새로고침 후 상태 유지" 검증은 이 작업이 끝나야 통과한다.

2. **Phase 3 약속은 로컬 저장으로 1차 구현 (필수)**
   `webapp/src/firebase.js`에는 Auth가 없고 공개 클라이언트 설정만 있다. Firestore 보안 규칙 없이 `pulseCommitments` 컬렉션을 열면 누구나 읽기·쓰기가 가능하다. 따라서 1차 구현은 §8.2 단서대로 **로컬 상태 저장**으로 한정하고 UI에 `이 기기 저장`을 명시한다. Firestore 전환은 인증·규칙 도입 이후 별도 작업으로 분리한다. 운영 배포 전 `pulseCommitments`를 Firestore로 올리지 않는다.

3. **engagementScore 일반화는 파서까지 손대야 함 (Phase 0)**
   `companyEngagement`의 `year === 2026 ? exOutlier2026` 예외(`pulseEngine.js:274`)를 제거하려면 `exOutlier2026` 단일 필드를 연도별 일반 필드로 바꿔야 한다. 그런데 `parseEngagementScore`(`pulseUpload.js:133`~)는 현재 연도 컬럼만 읽고 ex-outlier 변형 행은 읽지 않는다. 즉 **템플릿(`pulseTemplate.js`) + 파서 + 엔진 3곳을 함께** 수정해야 한다. Phase 0 작업 4(고정 비교 제거)에 이 범위를 포함한다.

4. **기존 저장 문서 백필 전략 (Phase 0)**
   §8A.5의 `schemaVersion`/`semanticKey`를 "지금부터 저장"해도 이미 저장된 `pulseResults/2024~2026` 문서에는 없다. 둘 중 하나를 명시적으로 선택한다: (a) 해당 연도 재업로드 유도, (b) 로드 시 메타데이터 누락 문서에 22문항 v1 기본 카탈로그를 주입하는 폴백. `comparisonPair(...)`가 메타 없는 과거 문서를 만나도 비교가 깨지지 않도록 폴백을 둔다.

5. **연도 카탈로그 로딩 교체 (Phase 0)**
   현재 `loadPulseYears`는 고정 배열을 `getDoc`으로 한 해씩 읽는다(`state.js:306`~`328`). §8A.2의 `getDocs(collection(db,"pulseResults"))` 카탈로그 방식으로 교체하고, `app.js`의 두 호출부(`app.js:2852`, `app.js:4186`)와 `pulseViews.js`의 `loadPulseYears(pulseYears(...))` 호출들도 함께 정리한다.
