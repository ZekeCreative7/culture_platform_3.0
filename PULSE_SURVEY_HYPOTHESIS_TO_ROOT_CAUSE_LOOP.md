# Pulse Survey Executive Report Structure

Date: 2026-07-04
Scope: Pulse Survey page improvement plan and visual report direction.

## 10-Loop Final Judgment

현재 Pulse Survey 페이지의 핵심 역할은 조직 상태를 확정 진단하는 것이 아니다.

핵심 역할은 넓은 조직 신호를 빠르게 감지하고, 그 신호를 바탕으로 원인 가설을 세우며, FGD와 IDI를 통해 핵심 원인을 좁히는 것이다. 핵심 원인이 확인된 뒤에야 전사 공통 세션, 본부별 워크숍, 팀장 코칭, 운영개선 프로그램을 설계한다.

다만 경영진 보고서의 첫 화면은 방법론으로 시작하면 안 된다. 먼저 현재 조직 상태를 짧고 명확하게 보여줘야 한다.

## Report Opening

### Better opening

후속조치 신뢰가 2년 연속 약화됐습니다.

### Supporting sentence

구성원은 의견을 내고 있지만, 그 의견이 처리되고 공유된다는 경험은 약해지고 있습니다. 원인은 아직 확정하지 않고, 본부별 경청으로 확인합니다.

### Data basis required

- 조사 연도와 조사 기간
- 응답자 수와 응답률
- 전년 동일 문항 비교 여부
- 본부별 표본 수
- 본부별 N이 작은 구간의 해석 제한
- 실제 데이터인지 예시 수치인지 표시

## Executive Report Flow

1. 핵심 판단
   - 지금 조직 상태를 한 문장으로 말한다.
   - 예: 후속조치 신뢰가 2년 연속 약화됐습니다.

2. 판단 근거
   - 현재값, 전년 대비 변화, 본부 간 격차를 함께 보여준다.
   - 숫자는 반드시 기준과 표본 제한을 붙인다.

3. 본부별 차이
   - 본부를 순위표로 만들지 않는다.
   - 우선 확인, 추가 확인, 다른 원인 가능성, 표본 검토로 구분한다.

4. 판단 보류
   - Pulse만으로 핵심 원인을 확정하지 않는다.
   - 리더 행동, 의사결정 구조, 업무 리듬, 후속조치 루틴 중 무엇이 반복 기제인지는 FGD/IDI로 확인한다.

5. 후속 조치 근거
   - 어느 본부부터 FGD를 할지 설명한다.
   - 어떤 민감 주제를 IDI로 분리할지 설명한다.
   - 본부장과 팀장이 먼저 닫아야 할 운영 루틴의 이유를 제시한다.

## Company To Division Logic

사장님은 전사 상태와 본부 간 차이를 함께 봐야 한다. 본부장은 자기 본부의 전년 변화, 전사 대비 차이, 본부 내 팀/직군 차이, 팀장 행동 과제를 봐야 한다.

권장 구조:

- 전사 핵심 신호
- 3년 흐름
- 본부별 확인 우선순위
- 본부장용 짧은 판단 카드
- 팀장 행동 과제
- 본부별 FGD 질문
- IDI 분리 조건
- 확인된 원인에 따른 개입 후보

## Division Card Template

각 본부 카드는 아래 순서를 고정한다.

- 현재 신호
- 전년 대비 변화
- 전사 대비 차이
- 가능한 해석
- 아직 모르는 것
- FGD 질문
- IDI 분리 조건
- 팀장 행동 과제
- 세션 또는 운영개선 후보

예시:

영업본부는 후속조치 신뢰가 전사보다 낮고 2년 연속 하락했습니다. 먼저 고객/목표 압박 속에서 제안 처리 결과가 어떻게 공유되는지 FGD로 확인합니다. 리더 반응이나 평가 불이익 경험은 IDI로 분리합니다. 확인 전까지 프로그램은 추천하지 않습니다.

## Cause Hypothesis Standard

가설은 설명이 아니라 검증 가능한 문장이어야 한다.

나쁜 문장:

- 몰입이 낮다.
- 리더십 문제가 있다.
- 심리적 안전감이 부족하다.

좋은 문장:

- 의견 제기 후 처리 결과가 공유되지 않아 후속조치 신뢰가 낮아졌을 가능성이 있다.
- 피드백이 업무 조정 전에 오지 않아 사후 지적으로 경험될 가능성이 있다.
- 반대 의견 이후 리더 반응이 다음 발언 의향에 영향을 줄 가능성이 있다.

각 가설은 반드시 아래를 포함한다.

- 관찰 신호
- 검증 가설
- 대안 설명
- 반증 기준
- FGD 질문
- IDI 분리 조건
- 확인 전 금지할 결론
- 확인 후 가능한 개입 유형

## FGD And IDI Rules

FGD는 점수 이유를 추궁하는 자리가 아니다. 공유된 경험 패턴과 조직 언어를 확인하는 절차다.

FGD 질문은 다음처럼 쓴다.

- 어떤 상황에서 의견을 냈습니까?
- 그 의견은 이후 어떻게 처리됐습니까?
- 처리 결과는 누구에게, 언제, 어떤 방식으로 공유됐습니까?
- 반대로 잘 처리됐다고 느낀 사례는 무엇입니까?

IDI는 아래 경우에 분리한다.

- 리더십, 평가, 관계 갈등처럼 개인 위험이 큰 주제
- 집단 토론에서 침묵될 가능성이 높은 주제
- 소수 직군 경험이 평균에 묻힐 가능성이 있는 주제
- FGD에서 해석이 크게 갈리는 주제

## Intervention Logic

개입 방식은 원인의 위치에 따라 달라진다.

- 인식과 관계 문제: 전사 공통 세션 또는 본부별 워크숍
- 리더 행동 문제: 팀장 코칭과 회의 운영 루틴
- 업무 구조 문제: 운영개선 과제
- 후속조치 루프 문제: You Said / We Heard / We Will / We Did 공유 루틴
- 소진과 회복 문제: 업무량 조정, 회복 리듬, 도움 요청 경로

프로그램은 원인 후보가 아니라 확인된 작동 기제에 연결한다.

## Report Language Standards

### Use

- 핵심 판단
- 판단 근거
- 관찰 신호
- 검증 가설
- 판단 보류
- 확인 우선순위
- 본부별 차이
- 집단 인터뷰(FGD)
- 개별 심층 인터뷰(IDI)
- 핵심 원인
- 원인 후보
- 확인된 원인
- 다음 조치
- 개입 후보
- 과잉 해석 위험

### Avoid

- 인사이트
- 액션
- 루트코즈 엔진
- 파이프라인
- 워크벤치
- 리서치 패킷
- 확정 진단
- 자동 원인 진단
- 프로그램 자동 추천
- 순위표처럼 보이는 본부 비교
- 출처 없는 수치

## Implementation Plan

### React implementation principle

이 개편은 React 화면으로 구현한다. 새 HTML을 그대로 붙이는 방식이 아니라, 현재 `PulsePage.jsx`와 `PulseComponents.jsx`의 React 구조 안에서 탭, 카드, 차트, 프롬프트 패널을 만든다.

핵심 원칙은 계산과 문구를 분리하는 것이다.

- 자동 계산: 연도별 업로드 데이터, 전년 대비, 본부별 차이, N/응답률, 우선 확인 본부, 3년 추이
- 사람 검토: 핵심 판단 문장, 원인 가설, FGD/IDI 질문, 실무 조치 근거
- 저장 대상: 업로드 원본 데이터, 계산 결과, GPT 프롬프트, 사용자가 붙여넣은 해석 문구

### React migration cautions

- `pulseEngine.js`를 계산의 중심 Module로 둔다. React 컴포넌트 안에서 본부 우선순위, 전년 대비, 가설 후보 계산을 중복 구현하지 않는다.
- `PulseComponents.jsx`는 표시와 상호작용을 담당한다. 업로드 파싱, DB 저장, 세션 연결 같은 구현을 화면 컴포넌트에 직접 넣지 않는다.
- `pulseUpload.js`는 연도별 데이터 계약을 지킨다. 업로드 템플릿이 바뀌면 parser, preview, 저장 payload, 테스트를 같이 바꾼다.
- 자동 분석 문장은 확정 진단처럼 보이면 안 된다. 화면에는 `관찰 신호`, `검증 가설`, `판단 보류`, `후속 확인`을 유지한다.
- 본부별 비교는 순위표로 보이지 않게 한다. 정렬은 가능하지만 라벨은 `우선 확인`, `추가 확인`, `다른 원인 가능성`, `표본 검토`로 둔다.
- 세션 만들기 연결은 원인 후보가 아니라 확인 절차를 넘긴다. `이 본부는 이 세션을 해야 함`보다 `이 신호를 확인하기 위한 세션/FGD 후보`로 전달한다.
- Report, Analytics, Dashboard가 같은 계산 결과를 재사용해야 한다. 화면마다 다른 계산식을 만들면 숫자가 달라져 신뢰가 깨진다.
- 팀 단위 화면은 Pulse 원본이 본부 단위라는 점을 표시한다. 팀은 `year + orgUnitId -> pulseDivisionId` 매핑을 통해 본부 결과를 상속한다.
- GPT 프롬프트 생성은 첫 구현에서 자동 세션 생성보다 우선한다. 자동 생성보다 `프롬프트 복사 -> GPT 결과 붙여넣기 -> 사용자 검토 후 저장`이 더 안전하다.
- 매년 데이터 업로드 후에는 브라우저에서 `업로드 -> 저장 -> 연도 선택 -> 경영 요약 변경 -> 본부별 화면 변경 -> 보고서/세션 연결`까지 확인한다.

### 1단계: 경영진 요약 화면

- 현재 Overview 상단을 경영진 보고서형으로 재구성한다.
- 포함: 핵심 판단, 데이터 기준, 핵심 신호 3개, 본부별 차이, 판단 보류, 후속 조치 근거.

### 2단계: 본부별 화면

- 본부별 카드에 현재값, 전년 대비, 전사 대비, N, 응답률을 표시한다.
- 본부장과 팀장이 볼 다음 조치를 분리한다.

### 3단계: FGD 질문 설계

- 가설별 FGD 목적, 참가자 구성, 질문, 반증 질문, 관찰 포인트를 만든다.

### 4단계: IDI 판단 메모

- 민감 주제를 IDI로 분리하는 조건과 공유 제한을 표시한다.

### 5단계: 원인 판단 메모

- 증상, 작동 기제, 조건, 개입 지점, 배제한 원인, 미해결 질문을 관리한다.

### 6단계: 개입 설계 연결

- 확인된 원인에 따라 전사 공통 세션, 본부별 워크숍, 팀장 코칭, 운영개선 과제로 연결한다.

## Success Criteria

첫 화면을 본 경영진이 60초 안에 아래 질문에 답할 수 있어야 한다.

- 지금 우리 조직의 핵심 신호는 무엇인가?
- 어떤 근거로 그렇게 판단하는가?
- 어느 본부에서 더 강하게 나타나는가?
- 아직 단정하지 말아야 할 원인은 무엇인가?
- 어떤 FGD/IDI를 왜 진행하려 하는가?
- 본부장과 팀장은 무엇을 먼저 닫아야 하는가?
- 원인이 확인되면 어떤 세션이나 운영개선으로 연결되는가?

## Additional Loop Decision

이번 10회 리뷰에서 결론은 충분히 수렴했다. 더 많은 아이디어 루프보다 실제 데이터 연결 전 최종 1회 검수만 필요하다.

검수 항목:

- 숫자 출처와 표본 기준
- 첫 화면의 문장 길이
- 본부별 비교가 순위표처럼 보이지 않는지
- FGD/IDI가 원인 검증 절차로 보이는지
- 프로그램 추천이 확인 전 결론처럼 보이지 않는지
