# 세션 페이지 UX 감사

작성일: 2026-07-09

## 결론

세션 페이지는 기능적으로는 강하다. 조직 DB 기반 대상 선택, 세션 유형별 기본 회차, 설문 질문 생성 프롬프트, 일정 캘린더, 설문 응답 업로드 상태, 정성 분석 진입까지 이미 갖고 있다.

하지만 처음 쓰는 사람 기준으로는 "여기가 무엇을 하는 화면인지"보다 "등록된 세션 목록"이 먼저 보인다. 현재 화면은 세션 운영의 전체 스토리보다 스케줄 관리와 카드 상태를 먼저 보여주기 때문에, 사용자가 다음 순서를 스스로 추론해야 한다.

세션 페이지의 핵심 메시지는 다음처럼 바뀌어야 한다.

> 조직 진단과 팀 이슈를 실제 세션으로 배정하고, 일정-설문-응답-리포트까지 운영 상태를 관리하는 화면

## 현재 화면의 실제 첫 인상

데스크톱 첫 화면:
- 상단: 운영 상태 패널, DB/배포/데이터 상태
- 페이지 헤드: `세션 운영`, `문화 세션 스케줄 관리`
- 주요 액션: `+ 새 세션`, 더보기 DB 관리
- 본문: `목록 / 일정 캘린더`, `등록된 세션 7개`
- 카드: 유형, 팀/기수, 일정 확정 수, 미확정 수, 설문 응답 업로드 단계, 설문 데이터 없음

모바일 첫 화면:
- 운영 상태 패널이 화면 상단 200px 이상을 차지한다.
- 세션 페이지의 제목은 첫 화면 중간 이후에 나온다.
- 사용자는 시스템 상태를 먼저 보고, 세션 페이지의 목적은 나중에 본다.

첫 화면에서 빠지는 것:
- 이 페이지가 전체 플랫폼 플로우에서 어디인지
- 세션을 왜 만드는지
- 세션 생성 후 무엇을 해야 하는지
- 카드의 경고를 어떻게 해결해야 하는지
- 세션 유형별 차이가 무엇인지

## 기능적 강점

1. 세션 유형이 제품 도메인에 맞게 분화되어 있다.
   - 팀빌딩: 한 팀 전체 참여
   - 리더십: 여러 팀 리더 그룹 구성
   - 협업: 리더십 세션 기반 또는 전체 조직 무작위 구성
   - 커스텀: 팀별/무작위/전사 범위 구성
   - 운영 서베이: 조직문화 밖 단발 설문

2. 조직 DB와 연결된 대상 선택이 강하다.
   - 부문/본부/팀을 고르면 리더와 구성원 정보를 자동으로 불러온다.
   - 수기 입력보다 운영 오류가 적다.

3. 세션 등록과 설문 설계가 연결되어 있다.
   - 드로어 안에서 `설문 질문 생성 프롬프트`를 제공한다.
   - Pulse 매핑이 있으면 진단 맥락을 설문 설계에 연결할 수 있다.

4. 카드가 운영 상태를 요약한다.
   - 시작전/진행중/완료
   - 일정 확정 회차
   - 미확정 회차
   - 설문 응답 업로드 단계
   - 정성 분석 액션

5. 캘린더와 출석 체크까지 이어진다.
   - 월/주/일 보기로 실제 운영 일정 확인이 가능하다.
   - 명단이 있는 세션은 일정에서 출석 체크로 들어갈 수 있다.

## 기능적 약점

1. 페이지 제목이 실제 기능보다 좁다.
   - 현재 제목은 `문화 세션 스케줄 관리`다.
   - 실제로는 스케줄뿐 아니라 대상 구성, 설문 설계, 응답 적재 상태, 분석 준비 상태까지 다룬다.
   - 이 때문에 사용자는 "일정표 화면"으로 오해할 수 있다.

2. 전체 여정이 보이지 않는다.
   - 실제 흐름은 `세션 등록 -> 설문 설계/배포 -> 응답 수집/업로드 -> 분석/리포트 -> 커뮤니케이션/실행 과제`다.
   - 세션 페이지에는 이 흐름을 보여주는 진행 바나 단계 안내가 없다.

3. 카드의 경고가 행동으로 이어지지 않는다.
   - `설문 데이터 없음`은 중요하지만, 카드 안에서 바로 `설문 만들기`, `응답 업로드`, `리포트 보기`로 이어지지 않는다.
   - 사용자는 사이드바에서 어디로 가야 할지 알아야 한다.

4. 세션 유형 선택이 초심자에게 어렵다.
   - 팀빌딩/리더십/협업/커스텀/운영 서베이가 있지만, 각각 언제 쓰는지 선택 전에 충분히 알려주지 않는다.
   - 특히 `협업`은 리더십 세션 기반이라는 전제가 있어 사전 지식이 필요하다.

5. 드로어가 너무 많은 일을 한 번에 한다.
   - 세션 유형, 기수, 연도, 대상 구성, 설문 프롬프트, 회차 일정이 한 화면 안에 들어간다.
   - 숙련자에게는 빠르지만 초심자에게는 "뭘 먼저 해야 하지?"가 생긴다.

6. 운영 상태 패널이 페이지 스토리보다 먼저 온다.
   - 모든 페이지에 공통으로 들어가는 시스템 상태 패널이 세션 페이지 첫 맥락을 밀어낸다.
   - 특히 모바일에서는 세션 화면 자체의 의미가 첫 화면에서 늦게 나타난다.

7. 관리용 액션이 초심자 첫 화면에 노출된다.
   - `DB 다운로드`, `DB 전송`, `JSON 백업`은 운영자/관리자 기능이다.
   - 처음 쓰는 사람에게는 주요 플로우보다 시스템 관리 기능이 먼저 보일 수 있다.

## 논리 점프 구간

1. "세션"이라는 단어의 의미 점프
   - 사용자에게 세션이 단순 일정인지, 조직문화 개입 단위인지, 설문/리포트의 기준 단위인지 설명되지 않는다.

2. Pulse에서 세션으로 넘어가는 점프
   - 홈과 진단 리포트는 조직 지원 신호를 말한다.
   - 세션 페이지는 바로 세션 목록을 보여준다.
   - "어떤 진단 신호를 어떤 세션으로 바꾸는가"가 빠져 있다.

3. 세션 등록에서 설문 생성으로 넘어가는 점프
   - 설문 프롬프트는 드로어 안에 있지만, 세션 등록 후 설문 페이지에서 무엇을 해야 하는지 연결되지 않는다.

4. 완료 상태와 데이터 없음의 충돌
   - 카드에 `완료`와 `설문 데이터 없음`이 동시에 보인다.
   - 운영 상태로는 완료일 수 있지만 분석 상태로는 미완료다.
   - 사용자는 이 세션이 끝난 것인지, 아직 처리해야 하는 것인지 혼란스러울 수 있다.

5. 협업 세션의 전제 점프
   - 협업 세션은 리더십 세션 기반으로 추천 팀을 가져올 수 있다.
   - 하지만 처음 보는 사람은 "왜 리더십 세션이 먼저 필요하지?"를 화면에서 알기 어렵다.

6. 응답 업로드와 설문 배포의 순서 점프
   - 카드에는 업로드 단계만 보인다.
   - 하지만 사용자는 먼저 설문을 만들고 배포해야 한다.
   - "설문이 없어서 업로드가 없는 것인지, 설문은 있는데 응답이 없는 것인지"가 분리되지 않는다.

## 개선 방향

### 1. 페이지 헤드라인을 역할 중심으로 바꾸기

현재:
- 문화 세션 스케줄 관리

권장:
- 조직 이슈를 세션으로 운영하기
- 또는: 세션 운영 보드

보조 문장:
- 진단에서 발견한 이슈를 팀빌딩, 리더십, 협업 세션으로 배정하고 일정, 설문, 응답, 리포트 준비 상태를 관리합니다.

### 2. 첫 화면에 운영 여정 4단계를 넣기

상단에 작은 단계 요약을 둔다.

1. 대상 정하기
2. 세션 일정 잡기
3. 설문 만들고 응답 모으기
4. 리포트와 실행 과제로 넘기기

이건 설명문이 아니라 현재 상태 요약이어야 한다.

예:
- 대상 미정 0개
- 일정 미확정 2회차
- 설문 미준비 7개
- 리포트 준비 0개

### 3. 카드에 다음 행동 버튼을 붙이기

카드 상태별 권장 액션:
- 일정 미확정: `일정 확정`
- 설문 없음: `설문 만들기`
- 응답 없음: `응답 확인`
- CSV 필요: `응답 업로드`
- 사전/사후 완료: `리포트 보기`
- 정성 응답 있음: `정성 분석`

카드는 상태 표시가 아니라 운영 큐가 되어야 한다.

### 4. 세션 유형 선택을 카드형 가이드로 바꾸기

드로어 첫 단계에서 유형별 사용 상황을 보여준다.

- 팀빌딩: 한 팀의 분위기와 협업을 회복/강화할 때
- 리더십: 여러 팀 리더가 같은 이슈를 다룰 때
- 협업: 리더십 세션 후 팀 간 실행 그룹을 만들 때
- 커스텀: 특정 대상만 따로 모아 운영할 때
- 운영 서베이: 문화 세션이 아닌 단발 점검을 할 때

초심자에게는 이 한 줄이 기능명보다 중요하다.

### 5. 드로어를 단계형으로 나누기

현재 드로어는 한 번에 너무 많이 보여준다. 추천 구조:

1. 세션 목적/유형
2. 대상 구성
3. 회차 일정
4. 설문 설계 준비

숙련자용으로는 한 화면 편집을 유지하되, 기본 첫 진입은 단계형이 더 안전하다.

### 6. `완료`를 운영 완료와 분석 완료로 분리하기

현재 카드의 `완료`는 일정 기준이다. 하지만 사용자는 완료를 전체 완료로 읽는다.

권장 상태:
- 일정 전
- 운영 중
- 운영 종료
- 응답 대기
- 분석 준비
- 리포트 완료

또는 카드 안에 두 줄로 분리:
- 운영: 완료
- 데이터: 설문 응답 없음

### 7. 모바일에서는 운영 상태 패널을 접기

모바일 첫 화면에서는 시스템 상태보다 페이지 목적이 먼저 보여야 한다.

권장:
- 정상 상태면 운영 상태 패널을 한 줄 칩으로 축소
- 문제가 있을 때만 펼쳐서 노출
- 세션 페이지 H1과 주요 액션을 첫 화면 상단으로 올리기

## 우선순위

P0. 첫 화면 스토리 수정
- 헤드라인, 보조 문장, 운영 여정 4단계, 핵심 상태 요약을 넣는다.
- 이유: 이게 없으면 초심자는 세션 페이지를 "일정 목록"으로 읽는다.

P1. 카드의 다음 행동 연결
- `설문 데이터 없음` 같은 경고를 버튼으로 바꾼다.
- 이유: 사용자가 사이드바 구조를 몰라도 다음 작업을 할 수 있어야 한다.

P1. 상태 모델 문구 정리
- `완료`를 일정 완료와 분석 완료로 분리한다.
- 이유: 현재는 완료인데 미완료처럼 보이는 모순이 있다.

P2. 세션 등록 드로어 단계화
- 유형 선택 가이드와 대상 구성 단계를 분리한다.
- 이유: 세션 종류가 많아질수록 초심자 선택 부담이 커진다.

P2. 모바일 첫 화면 재배치
- 운영 상태 패널을 축소하고 페이지 헤드를 위로 올린다.
- 이유: 모바일에서는 페이지의 목적이 너무 늦게 보인다.

P3. 관리자 기능 숨김
- DB/JSON 기능은 관리자 메뉴나 개발자 모드로 이동한다.
- 이유: 첫 사용자에게 핵심 플로우와 무관한 기능이 보인다.

## 추천 첫 번째 개발 슬라이스

코드 변경을 시작한다면 첫 슬라이스는 다음이 가장 좋다.

1. 세션 페이지 상단에 `세션 운영 보드` 스토리 밴드 추가
2. 운영 여정 4단계 상태 요약 추가
3. 세션 카드에 상태별 다음 액션 버튼 1개 추가
4. 모바일에서 운영 상태 패널 축소

이 슬라이스가 좋은 이유는 기능 로직을 크게 건드리지 않고, 첫 사용자 이해도와 다음 행동 문제를 바로 개선하기 때문이다.

## 확인 질문

세션 페이지의 1순위 사용자는 누구로 볼지 결정해야 한다.

추천 답: `조직문화 운영 담당자`

이 사용자는 진단 결과를 보고 세션을 만들고, 설문/응답/리포트 준비 상태를 놓치지 않아야 한다. 따라서 화면은 "세션 생성 폼"보다 "운영해야 할 다음 일"을 중심으로 재구성하는 것이 맞다.
