# 에이전트 파이프라인 — 다음 프로젝트 운영 가이드

## 에이전트 구성 요약

| # | 에이전트 | 파일 | 투입 시점 | 핵심 목적 |
|---|---|---|---|---|
| 01 | 범위·문제 정의 | AGENT_01_SCOPE.md | 기능 시작 전 | 완료 기준 + 모듈 우선순위 고정 |
| 02 | DB 아키텍처 | AGENT_02_DB.md | 데이터 구조 변경 전 | 스키마 설계 + 영향 범위 분석 |
| 03 | 디자인 기준 | AGENT_03_DESIGN.md | UI 작업 전 | 레퍼런스 1장 + 컴포넌트 명세 고정 |
| 04 | 구현 | AGENT_04_IMPL.md | 01~03 승인 후 | 모듈 순차 구현 |
| 05 | 코드 검토 | AGENT_05_CODE_REVIEW.md | 구현 직후, 커밋 전 | 작동·클린·효율·확장성·성능 |
| 06 | 서버 통신 검토 | AGENT_06_SERVER.md | API 변경 시, 출시 전 | Firestore 설계·부하·속도·실패 처리 |
| 07 | UX 검토 | AGENT_07_UX.md | 화면 완성 후 | 직관성·플로우·모바일·엣지케이스 |
| 08 | 메타 검토 | AGENT_08_META_REVIEW.md | 사이클 완료 후 | 에이전트 산출물 자체 품질 평가 |

## 표준 흐름

```
[새 기능 요청]
       ↓
  AGENT_01: 브리프 작성 (5분)
       ↓
  AGENT_02: DB 스키마 (DB 변경 시)
  AGENT_03: 디자인 명세 (UI 변경 시)
       ↓
  AGENT_04: 구현 (M1 → 검증 → M2 → 검증 ...)
       ↓
  AGENT_05: 코드 검토 → 🔴 수정 → 커밋
       ↓
  AGENT_06: 서버 통신 검토 (API 변경 시)
  AGENT_07: UX 검토 (UI 변경 시)
       ↓
  [출시]
       ↓
  AGENT_08: 메타 검토 (사이클 회고)
```

## 핵심 원칙 (이전 프로젝트 교훈)

1. **AGENT_01 없이 AGENT_04 투입 금지** — 브리프 없는 구현은 재작업의 시작
2. **DB 변경 시 AGENT_02 필수** — "나눠줘" 요청이 어느 레이어인지 먼저 확인
3. **UI 작업 시 레퍼런스 1장 필수** — 레퍼런스 없는 UI는 3회 재작업 원인
4. **AGENT_05 🔴 항목은 커밋 전에 반드시 수정**
5. **AGENT_08은 사이클마다 한 번** — 프로세스 자체를 개선하는 루프
