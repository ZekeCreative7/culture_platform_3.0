## 2. 이번 작업에서 잘한 것

### 2.1 사용자가 잘한 것

- **빠른 방향 전환**: Streamlit 포기 결정을 주저 없이 내렸다. 프로토타입에 집착하지 않은 것이 이후 10일 빌드의 근거가 됐다.
- **실제 화면 기반 피드백**: "QR 찍으면 404 에러 떠", "PDF 다운로드하면 중간이 잘려나와", "모바일에서 이런 에러 떠" — 코드가 아니라 화면 결과로 피드백했고, 스크린샷을 자주 첨부했다. 이것이 AI와의 협업에서 가장 빠른 해결 경로였다.
- **메타인지 능력**: 개발 중간에 "솔직히 리뷰해줘 (06-20)", "설문 분석 전문가라고 생각하고 냉정하게 분석해줘 (06-22)" 처럼 AI에게 비판적 관점을 주문했다. 단순 실행이 아니라 품질 검증을 요청한 것이다.
- **문서화 의지**: `CLAUDE.md`를 처음부터 만들고 유지하면서 다음 세션에 컨텍스트를 넘기는 구조를 잡았다.
- **산업 지식 직접 투입**: Engagement Score 34% 계산 근거, 고객솔루션본부 조직 구조, Pulse 표본수 N=642 같은 정보를 AI가 알 수 없는 도메인 지식을 직접 제공했다. 이런 인풋 없이는 분석 결과를 신뢰하기 어려웠을 것이다.
- **끝까지 완성**: 10일 동안 여러 창을 이어가며 포기하지 않고 완성했다.

### 2.2 AI/에이전트가 잘한 것

- **구현 속도**: 요청 후 보통 10~30분 내에 동작하는 코드를 만들었다. 특히 Firebase 연동, 인증 게이트, 정성 분석 파이프라인처럼 복잡한 기능도 한 세션 안에서 처리했다.
- **기존 코드 파악**: 세션이 바뀌어도 `CLAUDE.md`와 git 히스토리를 읽고 컨텍스트를 빠르게 복원했다.
- **UI/UX 해석**: "밝고 희망한 톤으로", "연결감이 떨어져", "더 직관적이지 않을까" 같은 추상적 피드백을 구체적인 코드 변경으로 정확하게 해석했다.
- **분석 정확도**: 06-22 데이터 위기 때 "실제 데이터 손실이 아니라 App Check 코드 버그"임을 정확히 진단했고, 06-22~23 분석 리뷰에서 표본 불일치, 가중평균, 단일 문항 표기 등 7가지를 모두 구현했다.
- **자동 커밋**: 개발 중 자동 커밋 메커니즘이 작업을 보존해줬다.

### 2.3 다시 써야 할 성공 패턴

| 패턴 | 왜 좋았나 | 다음에 적용할 상황 |
|---|---|---|
| 스크린샷 + 에러 메시지 첨부 | AI가 추측 없이 정확한 문제를 봤다 | 모든 UI 버그, 배포 문제 |
| "냉정하게 분석해줘" 주문 | AI가 칭찬 모드를 벗어나 실제 문제를 찾았다 | 기능 완성 후, 릴리즈 전 |
| CLAUDE.md 유지 | 세션 간 컨텍스트 전달이 됐다 | 모든 멀티세션 프로젝트 |
| 도메인 지식 직접 제공 | AI가 현실과 다른 가정을 하지 않았다 | 데이터 입력, 분석 기준 설정 |
| 한 vertical slice 완성 후 다음 | 중간 확인이 됐고 방향이 흔들리지 않았다 | 기능 추가 전반 |

### 가장 비싼 재작업 Top 3

1. **대시보드 UI 3회 전면 재작업** — 디자인 레퍼런스 없이 시작해서 매번 방향을 바꿨다. Coinbase-inspired UI 확정까지 아마 가장 많은 코드 라인이 버려졌다.
2. **코호트 구조 리팩터** — 세션 카드, Change/Report 탭, 필터, 분석 로직까지 연결된 구조를 통째로 바꿨다. 데이터 모델을 처음에 제대로 잡았다면 피할 수 있었다.
3. **분석 로직 7개 개선** — 처음 구현할 때 "측정/방법론" 관점이 없어서 출시 직전에 대형 개선을 해야 했다. 분석 기능은 처음부터 방법론 전문가 관점에서 설계해야 한다.

### 재작업을 줄이는 핵심 원칙

- 화면을 바꾸면 실제 화면으로 확인한다.
- 데이터 구조를 바꾸면 저장/불러오기/보고서까지 이어서 확인한다.
- PDF, 모바일, 배포처럼 브라우저 화면과 다른 결과물은 별도 산출물로 검증한다.
- 커밋 전에는 "내가 바꾼 것만 들어갔는지" 확인한다.
- 분석 기능은 방법론 설계 먼저, 코드 나중이다.

## 5. 에이전트 구성 리뷰

### 이번 프로젝트에서 실제로 쓴 방식

사실상 **단일 에이전트(Claude Code)** 하나가 모든 역할을 했다 — 기획, 구현, 검증, 릴리즈, 기록까지. 컨텍스트가 넘치면 새 창을 열고 "Continue from where you left off"를 보내는 식으로 이어갔다. 이것은 빠르게 진행하기에는 효율적이었지만, 다음과 같은 한계가 반복됐다:

- 세션이 바뀌면 AI가 이전 결정의 *이유*를 모른다 (무엇은 알아도 왜는 모른다)
- 검증을 위한 "다른 관점"이 없다 (구현한 에이전트가 검증도 하면 자기 코드를 과신하기 쉽다)
- 기획/범위 확인 단계 없이 바로 구현에 들어가서 재작업이 반복됐다

### 다음 프로젝트 권장 에이전트 구성

| 역할 | 언제 투입 | 맡길 일 | 산출물 |
|---|---|---|---|
| **1. 범위 에이전트** | 새 기능 시작 전 5분 | 목표, 사용자, 이번 포함/제외, 완료 기준, 검증 환경 | 1쪽 개발 브리프 |
| **2. 데이터 에이전트** | 데이터 모델·저장 방식이 바뀔 때 | source of truth, 저장/불러오기 흐름, 영향 범위 | 데이터 영향도 메모 |
| **3. 구현 에이전트** | 범위 확정 후 | 실제 코드 수정 (한 번에 하나의 vertical slice) | 코드 변경 |
| **4. 검증 에이전트** | 구현 직후 | PC, 모바일, PDF, 실제 데이터, 빈 데이터, 배포 상태 | 검증 로그 |
| **5. 릴리즈 에이전트** | 커밋/푸시 전 | 변경 파일 점검, 임시 코드 제거, 커밋 메시지, 푸시 확인 | 커밋/푸시 기록 |
| **6. 기록 에이전트** | 큰 단위 작업 종료 시 | 결정·마찰·교훈 정리 | 세션 회고 문서 |

> 모든 단계를 한 에이전트에서 다 할 수도 있다. 다만, **"한 단계가 끝났다"는 체크포인트를 사용자가 명시적으로 확인하는 습관**이 중요하다. 에이전트를 분리하든 통합하든, 체크포인트를 건너뛰면 재작업이 생긴다.

## 6. 다음 프로젝트 기본 프로세스

### 6.1 시작할 때 반드시 고정할 것

```
- 목표: "이 기능이 완성되면 어떤 상황이 달라지나?"

- 수정/신규 화면: 이번에 바뀌는 화면 목록
- 사용할 데이터: 어디서 오는가 (DB, 하드코딩, 계산)
- 이번 개발에 반드시 포함: (명확하게)
- 이번에는 제외: (명확하게, "나중에" 박스)
- 완료 기준: 화면에서 어떻게 나오면 완료인가
- 검증 환경: PC / 모바일 / PDF / 배포 중 무엇을 봐야 하는가
```

### 6.2 개발 중 지켜야 할 것

- 새 아이디어가 나오면 바로 넣지 말고 "이번 개발" 또는 "다음 개발"로 분류한다.
- 화면 변경은 PC와 모바일을 분리해서 본다.
- 데이터 변경은 저장, 표시, 분석, 내보내기까지 연결해 본다.
- 분석/리포트 기능은 방법론 설계(무엇을 어떻게 계산하는가) 먼저 결정한다.
- 배포/캐시 이슈가 있는 앱은 버전 문자열 또는 서버 상태를 항상 같이 확인한다.

### 6.3 완료라고 말하기 전 체크리스트

- [ ] 목표가 처음 정의한 완료 기준을 만족한다.
- [ ] 주요 화면에서 직접 확인했다.
- [ ] 모바일 또는 특수 출력물이 있으면 별도 확인했다.
- [ ] 저장/불러오기 흐름이 있다면 실제 데이터로 확인했다.
- [ ] 불필요한 임시 파일/테스트 코드가 남지 않았다.
- [ ] 변경 파일을 확인했다.
- [ ] 커밋/푸시가 필요한 경우 완료했다.
- [ ] 이번 작업의 교훈을 짧게 남겼다.

## 7. 최종 결론

### 이번 플랫폼 개발을 한 문장으로 요약하면

> 사용자의 빠른 결단력과 도메인 지식, AI의 구현 속도가 합쳐져 10일 만에 프로덕션급 플랫폼을 만들었다. 다만 데이터 모델과 분석 방법론이 처음에 정해지지 않아서 중반부의 재작업이 30% 이상 차지했다.

### 가장 잘한 선택

- **Streamlit 즉시 포기** (06-15): 프로토타입에 미련을 두지 않았다. 이 결정이 없었다면 지금 이 플랫폼은 없었다.
- **"냉정하게 분석해줘" 요청** (06-22): 출시 직전에 스스로 전문가 리뷰를 주문한 것이 분석 품질을 한 단계 올렸다.

### 가장 아쉬운 선택

- **대시보드 UI를 레퍼런스 없이 시작한 것**: 3번의 전면 재작업 중 2번은 레퍼런스 1장만 먼저 정했어도 막을 수 있었다.
- **코호트 데이터 모델을 코딩 전에 그리지 않은 것**: "팀빌딩 1기 안에 팀장이 왜 있지?" 혼란이 대형 리팩터로 이어졌다.

### 다음에는 이렇게 하겠다

1. **5분 브리프 먼저**: 기능 시작 전에 목표/포함/제외/완료 기준을 1쪽으로 고정한다.
2. **레퍼런스 스크린샷 1장**: 새 UI 작업은 레퍼런스 화면 1장을 붙이고 시작한다.
3. **데이터 모델 먼저 그리기**: 저장/불러오기/분석/출력 흐름을 한 장에 그린 뒤 코딩한다.
4. **분석 기능은 방법론 먼저**: 무엇을 어떻게 측정하는지를 결정한 뒤 코드로 옮긴다.
5. **검증 체크포인트**: PC → 모바일 → PDF/출력물 순서로 반드시 확인한다.

## 8. 다음 프로젝트 운영 원칙 10개

1. **완료 기준 먼저**: "어떻게 나오면 완료인가"를 코딩 전에 한 문장으로 정한다.
2. **레퍼런스 1장**: 새 UI는 레퍼런스 화면 1장을 먼저 공유한다.
3. **데이터 모델 먼저**: 저장/불러오기/보고서 흐름을 먼저 그린다.
4. **한 slice씩**: 한 번에 하나의 기능만 만들고, 만든 것을 확인한 뒤 다음으로 넘어간다.
5. **화면으로 확인**: 코드 리뷰가 아니라 실제 화면으로 확인한다. 스크린샷을 붙인다.
6. **모바일/PDF 별도 체크**: 브라우저 화면이 좋아도 모바일과 PDF는 별도로 본다.
7. **도메인 지식은 직접**: AI가 모르는 업무 지식은 숫자와 구조로 명확하게 전달한다.
8. **"냉정하게 리뷰" 주문**: 완성 후 릴리즈 전에 전문가 관점 리뷰를 한 번 요청한다.
9. **세션 시작 3줄 요약**: 새 창을 열 때 "지금까지 된 것 / 이번 목표 / 완료 기준"을 3줄로 제공한다.
10. **재작업은 기록**: 무언가를 되돌렸다면 이유를 한 줄로 남긴다. 같은 실수를 반복하지 않는다.

## 10. 이 회고 작성 자체에 대한 메타 회고

### 잘된 점
- JSONL 세션 파일 9개를 직접 읽어 사실 기반으로 작성했다 (추측 최소화)
- `feedback/sessions/` 폴더에 세션별 회고 → `FINAL_INTEGRATED_REVIEW.md` 통합이라는 구조가 실제로 작동했다
- 사용자가 "여러 창에서 작업 후 회고"라는 목표를 일찌감치 설정하고 `SESSION_REVIEW_TEMPLATE.md`까지 준비해둔 덕분에 통합이 빠르게 됐다

### 아쉬운 점
- 세션 회고 파일을 "세션 종료 직후"가 아닌 프로젝트 완성 후에 한꺼번에 만들었다 — 생생한 기억보다는 git 로그 재구성에 의존해야 했다
- 일부 세션의 세부 결정 이유(왜 그 방향을 선택했는가)가 커밋 메시지만으로는 복원되지 않았다

### 다음 프로젝트에서 회고를 더 잘하는 방법
- **큰 기능 하나가 끝날 때마다** `sessions/` 파일 하나를 바로 작성한다 (5분이면 충분)
- 커밋 메시지에 **"왜"** 를 더 넣는다 ("어떻게"가 아니라 "왜 이 방향을 선택했는지")
- 재작업·revert 발생 즉시 한 줄 메모를 남긴다

## 12. 핵심 기술 교훈 (재사용 가능한 코드 패턴)

> 추상적 프로세스 교훈이 아니라, 실제 버그와 수정 코드에서 나온 구체적 패턴이다.

### T-1. Firebase App Check 디버그 토큰 영속화

**문제**: `self.FIREBASE_APPCHECK_DEBUG_TOKEN = true` 사용 시 → 매 새로고침마다 새 UUID 생성 → 미등록 토큰 → Firestore 전체 오프라인. 수 시간의 "데이터 사라짐" 패닉으로 이어졌다.

```js
const STORAGE_KEY = 'FIREBASE_APPCHECK_DEBUG_TOKEN';
let token = localStorage.getItem(STORAGE_KEY);
if (!token) { token = crypto.randomUUID(); localStorage.setItem(STORAGE_KEY, token); }
self.FIREBASE_APPCHECK_DEBUG_TOKEN = token;
```

**적용 시점**: Firebase 연동 Day 2에 반드시. 기능 개발 전 인프라 완성 단계에서.

---

### T-2. Firestore `setDoc` undefined 필드 거부 패턴

**문제**: optional 필드에 `undefined`가 들어가면 `Unsupported field value: undefined` 에러.

```js
// 나쁜 예 — sessionType이 없으면 undefined 전달
{ sessionType: normalizeSessionType(survey.sessionType) }

// 좋은 예 — 없으면 키 자체를 제외
...(survey.sessionType ? { sessionType: normalizeSessionType(survey.sessionType) } : {})
```

**적용 시점**: Firestore write 코드 작성 시 모든 optional 필드에 적용.

### T-3. 매칭 로직은 공유 함수로 — `rowMatchesSurvey` 패턴

**문제**: `surveyRows()`의 응답 매칭 로직(surveyId OR sessionId+phase+cohort 폴백)을 고아 스캔 함수에서 별도 구현 → 기준 불일치 → 중복 카드 생성.

**패턴**: 같은 판단 기준을 두 곳 이상에서 쓰면 → 반드시 함수 추출 + 양쪽 재사용.

**확인 질문**: "이 기능은 기존 로직과 같은 판단 기준을 쓰는가?" — 새 기능 작성 전에 먼저.
