# 현재까지 확인된 작업 흐름 요약

> 이 문서는 모든 채팅창을 직접 읽은 최종 기록이 아니다.
> 현재 저장소 상태와 기억된 작업 흐름을 바탕으로, 최종 회고를 시작하기 위한 “초기 재료”다.
> 정확한 최종 판단은 `feedback/sessions/*.md`가 모인 뒤 다시 통합해야 한다.

---

## 확인된 주요 작업 축

### 1. 분석/리포트 구조 정리

- 분석 페이지는 특정 세션 소개 페이지가 아니라, 여러 세션 결과를 선택해 보는 분석 허브로 유지해야 한다는 방향이 잡혔다.
- 세션별 제목/설명은 페이지 최상단보다 선택된 결과 영역 안에 배치하는 쪽이 맞다는 교훈이 있었다.
- PDF 다운로드에서는 브라우저 화면과 출력물의 레이아웃이 다르게 깨질 수 있어, 실제 PDF 렌더링 검증이 필요하다는 점이 드러났다.

### 2. 대시보드와 운영 화면 고도화

- Home은 단순 안내 화면보다 실제 운영 대시보드처럼 작동해야 한다는 방향이 잡혔다.
- “오늘 할 일”, 액션 큐, 세션 편집 바로가기처럼 사용자가 지금 해야 할 일을 먼저 보여주는 쪽이 유효했다.
- 카드/액션 영역이 복잡해질 때는 기능을 없애기보다, 메뉴나 접힘 구조로 시각적 복잡도를 줄이는 방식이 잘 맞았다.

### 3. 모바일/반응형 보정

- 모바일에서는 PC 레이아웃을 단순 축소하면 정보가 겹치거나 우선순위가 흐려질 수 있었다.
- 로그인 화면, 조직도 카드, Home 순서처럼 모바일 전용 확인이 필요한 지점이 반복적으로 나왔다.
- 모바일 이슈는 스크린샷 기반 피드백이 특히 효과적이었다.

### 4. 데이터·Firestore·저장 흐름

- 조직도, 세션, 설문, 분석 결과가 서로 연결되면서 source of truth를 먼저 정하는 것이 중요해졌다.
- 저장/불러오기/새로고침/보고서 반영까지 연결해서 봐야 “작동한다”고 볼 수 있다.
- 데이터 구조가 바뀌는 작업은 UI 구현보다 앞에서 영향 범위를 확인해야 한다.

### 5. 정성/정량 분석과 Pulse 흐름

- 설문 진행률, 응답 분포, 적용형 필터처럼 운영자가 실제로 판단할 수 있는 정보가 중요했다.
- 정성 분석과 정량 데이터는 섞어 보이기보다 역할을 분리하고, 화면에서 어디에 표시되는지 명확히 해야 했다.

---

## 반복해서 나온 좋은 패턴

- 화면을 보고 즉시 피드백한다.
- 너무 큰 기능은 한 번에 끝내기보다, 작은 단위로 만들고 바로 확인한다.
- 사용자가 “이렇게 나와”라고 실제 스크린샷을 주면 문제 해결 속도가 빨라진다.
- 코드 변경 후 실제 브라우저/PDF/모바일 산출물로 확인하면 재작업이 줄어든다.
- 커밋/푸시 여부를 명시적으로 확인하면 작업 상태가 덜 꼬인다.

---

## 반복해서 나온 위험 패턴

- “분석 페이지”, “세션 결과”, “보고서”처럼 비슷한 개념의 경계가 흐려질 때 방향이 틀어질 수 있다.
- 캐시나 버전 문자열 때문에 실제 화면이 최신 코드와 다르게 보일 수 있다.
- 데스크톱에서는 좋아 보여도 모바일이나 PDF에서 깨질 수 있다.
- 새 요청이 들어올 때 이번 개발 범위가 계속 커질 수 있다.
- 검증 기준 없이 “완료”라고 말하면 바로 다음 피드백에서 문제가 드러날 수 있다.
