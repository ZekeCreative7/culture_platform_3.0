# 다음 개발을 위한 권장 에이전트 워크플로우

이 문서는 Culture Platform 3.0 개발 과정을 바탕으로, 다음 큰 기능이나 다음 프로젝트를 진행할 때 어떤 순서와 어떤 역할의 에이전트를 쓰면 좋은지 정리한 운영안이다.

---

## 핵심 원칙

빠르게 가되, 시작점과 완료 기준은 먼저 고정한다.  
한 번에 모든 것을 만들기보다, 검증 가능한 한 조각을 완성하고 그다음 확장한다.

---

## 권장 에이전트 역할

| 에이전트 | 목적 | 언제 쓰나 | 결과물 |
|---|---|---|---|
| 1. 제품/범위 에이전트 | 무엇을 만들지 좁힌다 | 기능 시작 전 | 개발 브리프 |
| 2. 구조/데이터 에이전트 | 데이터 흐름을 흔들리지 않게 한다 | 저장/분석/보고서가 얽힐 때 | 데이터 영향도 정리 |
| 3. UI/UX 에이전트 | 사용자가 보는 화면의 위계를 잡는다 | 화면이 복잡하거나 모바일이 중요할 때 | 화면별 수정 기준 |
| 4. 구현 에이전트 | 실제로 만든다 | 범위가 정리된 뒤 | 코드 변경 |
| 5. 검증 에이전트 | 진짜 작동하는지 본다 | 구현 직후, 수정 직후 | 검증 기록 |
| 6. 릴리즈 에이전트 | 안전하게 마무리한다 | 커밋/푸시 전 | 변경 파일 점검, 커밋/푸시 |
| 7. 기록/회고 에이전트 | 다음에 쓸 교훈을 남긴다 | 큰 단위 작업 종료 후 | 세션 회고 |

---

## 추천 진행 순서

### 1단계 — 10분 개발 브리프

제품/범위 에이전트에게 아래를 정리시킨다.

- 목표
- 사용자
- 수정할 화면
- 사용할 데이터
- 이번 개발에 포함
- 이번에는 제외
- 완료 기준
- 검증 환경

이 단계가 길어지면 안 된다. 목적은 회의가 아니라, 구현 중 흔들리지 않을 기준선을 만드는 것이다.

### 2단계 — 구조 확인

데이터가 조금이라도 얽히면 구조/데이터 에이전트를 먼저 쓴다.

- 기준 데이터는 어디인가?
- 화면은 어떤 데이터를 읽는가?
- 저장하면 어디에 남는가?
- 보고서/PDF/분석 화면까지 영향이 있는가?

### 3단계 — 구현

구현 에이전트는 한 번에 하나의 vertical slice만 맡긴다.

좋은 요청 예:

> “분석 결과 PDF가 A4 세로 기준으로 잘리지 않게 수정하고, 실제 PDF를 열어 검증한 뒤 커밋 전 상태까지 보고해줘.”

나쁜 요청 예:

> “분석 페이지 전체를 더 좋게 해줘.”

### 4단계 — 검증

검증 에이전트는 구현 에이전트와 다른 관점으로 본다.

- PC 화면
- 모바일 화면
- 실제 데이터
- 빈 데이터
- PDF/다운로드 산출물
- 저장/새로고침 후 유지 여부
- 배포/캐시 상태

### 5단계 — 릴리즈

릴리즈 에이전트는 마지막에 아래만 본다.

- 의도한 파일만 바뀌었는가?
- 임시 파일이 남았는가?
- 검증 기록이 있는가?
- 커밋 메시지가 이해 가능한가?
- 푸시가 실제로 됐는가?

### 6단계 — 회고

기록/회고 에이전트가 `feedback/sessions/NN-주제.md`를 만든다.

이때 중요한 것은 결과가 아니라 과정이다.

- 왜 이 결정을 했나?
- 어디서 헷갈렸나?
- 무엇을 되돌렸나?
- 다음에는 어떤 질문을 먼저 해야 하나?

## 다음부터 기본으로 쓸 요청문

```text
이번 개발을 시작하기 전에 먼저 10분 브리프를 잡아줘.
목표, 사용자, 수정할 화면, 사용할 데이터, 이번 개발에 포함할 것, 제외할 것, 완료 기준, 검증 환경을 짧게 정리하고,
애매한 부분은 합리적으로 가정해서 진행해.
구현 후에는 실제 화면/데이터 기준으로 검증하고, 커밋/푸시가 필요하면 마지막에 물어봐.
```

---

## 피해야 할 흐름

- 화면을 보지 않고 코드만 보고 완료 처리하기
- 새 아이디어를 전부 “이번 개발”에 넣기
- 데이터 출처를 확인하지 않고 UI부터 만들기
- 모바일/PDF/배포처럼 깨지기 쉬운 산출물을 마지막에 처음 확인하기
- 커밋 전에 임시 파일과 변경 범위를 확인하지 않기
