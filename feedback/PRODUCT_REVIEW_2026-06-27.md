# Culture Platform 3.0 — 제품 리뷰
**작성일:** 2026-06-27  
**관점:** 프로덕트 전문가 + 코드 실증 분석 + 기획 프레이밍

---

## 1. 현재 플랫폼 요약

**미션:** 한 명의 OD 실무자가 금융사 조직문화 개입 프로그램을 설계·운영·측정하고, 그 결과를 경영진에게 데이터로 제시하는 운영 도구.

**현재 구조:**
```
Pulse Survey (22문항, 글로벌 기업 운영)
    ↓ 도메인 분류 (심리적안전감 / 사일로해소 / 회복탄력성 / 전반분위기)
    ↓ RAG 위험 진단 + 세션 유형 추천
세션 등록 및 운영 (팀빌딩 / 리더십 / 협업)
    ↓ 사전·중간·사후 설문 업로드 (CSV)
문항별 응답 분석 + 질적 시그널
    ↓
변화 리포트 (레이더 차트, 차원별 점수)
```

**목적에 대한 판단:** 뼈대와 방향은 맞다. 하지만 지금은 "분석 결과를 보는 도구"이고, 목적인 "실무자가 다음 행동을 아는 도구"가 되려면 TO-DO 레이어와 데이터 영속성이 가장 먼저 해결되어야 한다.

## 2. Pro / Cons

### ✅ Pro

| 항목 | 평가 |
|---|---|
| 진단→개입→측정 파이프라인 | 구조 자체는 OD 방법론적으로 올바름 |
| 도메인 분류 (4개) | 과도하지 않고 실무적으로 설명 가능한 수준 |
| 세션 유형 추천 엔진 | Pulse 진단 결과에서 세션 타입을 자동 제안하는 로직은 경쟁 제품에 없는 실용적 기능 |
| N<3 마스킹 | 데이터 윤리 및 신뢰도 관리 잘 되어 있음 |
| 외부 벤치마크 비교 | Medallia / Chubb APAC 기준 제시 = 경영진 설득 레버 |
| 질적 시그널 분석 | `qual-signal.js` 존재 자체가 차별점 |
| 사전·사후 구조 | 방법론 상 타당한 변화 측정 설계 |

### ❌ Cons — 핵심 문제 5가지

**1. "할 일 알려준다"는 목적이 구현되어 있지 않음 (가장 심각)**

플랫폼의 1번 목적이 "내가 할 일을 알려준다"인데, 현재 플랫폼은 100% 후행(retrospective) 시각화 도구다. "다음 주에 뭐 해야 하나"를 알려주는 기능이 없다. 세션 추천은 타입(팀빌딩/리더십)만 나오고, 실제 실행 준비사항·팔로우업 일정·후속 설문 발송 시점 등 실무자 TO-DO는 없다.

**2. Pulse Survey 의존 구조의 취약성**

글로벌 Pulse Survey(22문항)는 사용자가 통제할 수 없는 외부 시스템이다.
- 글로벌 HR이 문항 번호나 척도를 바꾸면 도메인 로직 전체가 깨짐
- 22문항이 OD 세션 효과 측정용으로 설계된 게 아님 (참여도/만족도 측정 목적)
- Pulse와 세션 설문(사전/사후)이 별개 시스템이라 두 데이터 간 통계적 연결 없음
- 코드에 하드코딩: `"심리적안전감": [5, 17, 19]` — 문항 번호 변경 시 전체 도메인 분석 붕괴

**3. 점수/가중치의 과학적 신뢰도 미검증**

```js
// webapp/src/pulse/pulseEngine.js
const W = { level: 0.5, unfav: 0.35, decline: 0.15 };
const T = { managerLow: 0.55, orgGap: 0.10 };
```

이 수치들은 임의로 설정된 것이다. 경영진이 "이 위험 점수는 어떻게 계산되나요?"라고 물으면 답하기 어렵다. 실제로 이 가중치로 "위험"으로 분류된 부서가 개입 후 개선됐는지 검증된 데이터가 없다.

**4. 경영진 설득용 인과 서사가 없음**

데이터는 있지만 스토리가 없다.
- "A팀 세션 후 심리적안전감 +0.4점" — 이게 유의미한 변화인지 맥락이 없음
- 개입 안 한 부서와의 비교(대조군)가 없음
- 시계열 트렌드("2024년 대비 전체 조직문화 지수 변화")가 없음
- PDF 리포트가 세션별 스냅샷이라 "연간 누적 효과"를 보여주는 뷰가 없음

**5. localStorage = 데이터 소실 위험**

모든 세션·분석·응답 데이터가 브라우저 localStorage에 있다. 브라우저 캐시 삭제, 기기 교체, 크롬 업데이트 시 전부 사라진다. 경영진 보고를 위한 증거 데이터가 이 상태로 저장되는 건 심각한 운영 리스크다.

## 3. 설문 구조 분석 — 센싱 레이어의 문제

### 현재 설문 구조 평가

**Pulse 22문항:** 연 1회 기초체력 진단으로는 괜찮다. Q17(두려움 없이 이슈 제기), Q19(설문 후 조치), Q20(포용), Q21(소속), Q22(협업)은 변화 트래커 방향과도 잘 맞는다. 단, 너무 크고 느리다 — 연 1회 건강검진이라 "이번 주 개선되고 있나"를 느끼게 하기 어렵다.

**세션 사전/사후 문항:** 심리적안전감 3개, 사일로 3개, 회복탄력성 1개, 전반분위기 1개 (`webapp/src/utils.js:156` 기준). 세션 전후 비교에는 쓸 수 있지만 **"무엇이 실제로 바뀌고 있는지"를 잡기엔 얇다.**

### 부족한 센싱 5가지

1. **행동 변화** — 현재 문항은 느낌/인식 중심. "실제로 회의에서 말이 늘었는가", "부서 간 요청이 쉬워졌는가", "리더가 약속한 행동을 했는가"가 없다.

2. **전진 감각** — 운영자와 구성원이 "뭔가 조금씩 나아지고 있다"고 느끼는지 묻는 문항이 없다.

3. **약속 이행 체감** — Q19는 Pulse에 있지만 너무 큰 문항. 세션 단위로 "이번 세션에서 나온 의견이 후속조치로 이어지고 있다"를 따로 물어야 한다.

4. **변화 지속성** — 사후 설문만 있으면 세션 직후 만족도에 가까워진다. 30일/90일 후 "아직 행동이 남아 있는지"를 봐야 한다.

5. **운영 마찰** — 금융사 조직에서 중요한 건 감정만이 아니라 승인, 의사결정, 업무 인계, 우선순위 충돌 같은 구조적 마찰. 이를 직접 묻는 문항이 약하다.

### 올바른 설문 레이어 구조

| 레이어 | 역할 | 주기 |
|---|---|---|
| **Pulse Survey** | 연 1회 기초체력 진단 / 경영진 보고 / 리스크 탐지 | 연 1회 (현행 유지) |
| **세션 사전·사후 설문** | 개입 전후 변화 신호 측정 (행동 변화 문항 추가) | 세션별 |
| **30일/90일 Follow-up 설문** | 행동 변화 유지 여부 확인 | 세션 후 30일, 90일 |

### 추가 권장 문항 예시

**행동 변화 측정 (객관식):**
- "이번 세션 이후 우리 팀이 바로 시도할 행동이 명확하다."
- "세션에서 나온 의견이 실제 후속조치로 이어질 것이라고 믿는다."
- "최근 우리 팀에서는 불편한 이슈를 이전보다 더 안전하게 말할 수 있다."
- "리더가 세션에서 약속한 행동을 실제로 보여주고 있다."
- "부서 간 협업에서 막히는 지점이 이전보다 더 명확해졌다."
- "우리 팀은 작은 변화라도 실제로 만들 수 있다는 감각이 있다."
- "지난 2주 동안 팀/리더/조직에서 긍정적인 변화 신호를 본 적이 있다."

**정성 문항 (주관식, 현재 "도움이 된 점/전하고 싶은 메시지" 대체):**
- "세션 이후 실제로 달라진 작은 장면이 있다면 무엇입니까?"
- "아직 바뀌지 않았지만 꼭 후속조치가 필요한 것은 무엇입니까?"
- "리더나 회사가 지키면 신뢰가 생길 작은 약속은 무엇입니까?"

## 4. 지표 체계 재설계

지금은 Pulse와 세션 결과 중심인데, 지표를 4층으로 나눠야 한다. **Outcome이 늦게 움직이기 때문에 Leading Signal을 먼저 보여줘야 한다** — 다이어트에서 체중보다 먼저 수면·컨디션·운동 수행력을 보는 것과 같다.

| 층 | 지표 예시 |
|---|---|
| **Input** | 세션 수, 약속 등록 수, 커뮤니케이션 수 |
| **Process** | 참여율, 응답률, 약속 이행률, 후속 공유율 |
| **Leading Signal** | 발언 안전감, 정성 감정 변화, 리더 행동 변화, 협업 신호 |
| **Outcome** | Pulse 변화, 사전/사후 점수 변화, 리스크 감소, 조직별 회복 추세 |

## 5. 개선 우선순위

### 기획 변경

**[P0] 데이터 영속성 — Firebase 마이그레이션**  
`firebase.js`가 이미 있다. localStorage 위에 기획 기능을 쌓으면 데이터 날리고 처음부터 다시 하게 된다. 모든 기획 변경의 전제조건.

**[P1] Home을 "변화 진행감 트래커"로 재편**  
현재: 전체 부서 RAG 상태 테이블 (보는 것)  
목표: "오늘 할 일", "이번 달 조직문화 변화 진행률", 조직별 단계 상태

조직별 단계 예시:
```
진단 완료 → 세션 진행 중 → 약속 등록 → 사후 설문 완료 → 변화 신호 확인 → 정체 경고
```

**[P2] 세션을 변화 가설 기반으로 재설계**  
세션 생성 시 추가 항목:
- 이 조직의 현재 리스크 (Pulse 자동 연동)
- 이번 세션의 변화 가설
- 바꾸고 싶은 행동
- 30일 후 확인 항목

세션이 이벤트가 아니라 **변화 실험**이 된다.

**[P3] Change Evidence Report 신설**  
현재 리포트: 세션별 스냅샷 ("운영 결과")  
목표: 경영진이 받았을 때 바로 읽을 수 있는 1-페이지 보고서

포함 내용:
- 현재 핵심 조직 리스크
- 왜 이 리스크가 중요한가
- 어떤 개입을 했는가 (참여율 포함)
- 어떤 선행 변화가 보이는가
- 다음 의사결정은 무엇인가

**[P4] 30/90일 Follow-up 설문 구조 추가**

**[P5] 조직별 변화 타임라인**  
```
2026 Pulse 리스크 감지 → 경청 세션 진행 → 리더 약속 등록 → 1차 실행 완료 → 사후 설문 변화 → 30일 후 재확인
```

### 개발 변경

| 우선순위 | 항목 |
|---|---|
| 긴급 | Firebase Firestore 마이그레이션 (`firebase.js` 이미 준비됨) |
| 중요 | Pulse 도메인 매핑 버전 관리 (하드코딩 → 버전별 config) |
| 중요 | 세션 상태 머신 명확화 (생성됨/진행중/완료/보류) |
| 개선 | 가중치 설정 파일 분리 + 근거 주석 (경영진 질문 대비) |
| 개선 | 약속 보드와 세션 결과를 조직별 타임라인으로 연결 |
| 개선 | 지표마다 데이터 신뢰도(N-size) 표시 강화 |

### UX 변경

- **세션 카드:** 사전/사후 설문 완료 상태를 카드에서 바로 보이게
- **대시보드:** "점수 변화 없음 / 선행 신호 있음 / 후속 필요" 상태 판정 추가
- **리포트 PDF:** 설명 텍스트 + 인사이트 요약 자동 포함 (데이터 덤프 → 보고서)

## 6. 핵심 요약

| 항목 | 현재 | 목표 |
|---|---|---|
| 목적 1 (할 일 알려주기) | ❌ 없음 | Home TO-DO + 세션 체크리스트 |
| 목적 3 (경영진 설득) | ⚠️ 데이터만 있음 | Change Evidence Report |
| 목적 4 (팔로우업) | ⚠️ 수동 의존 | 30/90일 자동 트리거 |
| 데이터 안전성 | ❌ localStorage | Firebase |
| 변화 센싱 깊이 | ⚠️ 인식 중심 | 행동 변화 + Leading Signal |
| 플랫폼 정체성 | 분석하고 운영하는 도구 | 조직문화 변화가 축적되는 시스템 |

**한 문장으로:** 이 플랫폼은 이제 '무엇을 했는지'가 아니라 '조직이 어떻게 움직이고 있는지'를 보여줘야 한다.
