## 2. 이 창에서 달성한 것
- **5기 유령 기수 제거 + 리스크관리팀 정성 누락 해결**: 근본 원인은 리스크관리팀 사후 설문의 `sessionCohort=5`(세션 실제 2기)인 cohort drift. 세션을 기수의 단일 진실원천으로 삼아 로드 시 자동 보정.
- **세션 중심 매칭으로 통일**: 정량/정성/차트가 `cohort` 필드가 아니라 `sessionId`(+phase) 기준으로 매칭 → 기수 값이 어긋나도 데이터가 사라지지 않음.
- **시점(사전/중간/사후→사전/사후) 탭** 추가 후 중간 제거.
- **정성에 정량 누수 차단**: 시점별로 주관식 문항을 판정(다른 시점의 정의가 새지 않게) + 척도 보기 텍스트(`그렇다` 등)는 정성에서 제외(`isQualText`).
- **"객관식 없음" 표기**: 주관식 전용 설문(예: DT기획팀 중간)에서 가짜 응답수/빈 분포 대신 명확 표기.
- **q11 척도 환산**: 추천 매핑(그렇다4·조금그렇다3·보통이다3·아니다2)으로 점수화(`scoreOf`).
- **기수를 세션 유형별로 스코프**(유형→기수→세션): 팀빌딩 1기와 리더십 1기가 섞이던 문제 해결.
- **데이터 리셋 버튼**을 Change에서 제거, 설문/QR 화면에만 유지(확인 팝업 포함).
- **설문 실시간 동기화**, **업로드한 사전 데이터 기본 노출**, **중복 사전 설문 삭제**.

## 6. 사용자가 개선할 점 (해석)
- **데이터 입력 위생**: 다수 "버그"가 실은 데이터/설정 불일치였음 — 설문 `sessionCohort` 드리프트(5↔2), q11 라벨-데이터 불일치(주관식 칸에 척도값), 사전 설문 중복, 유형 혼합 기수. 입력 규약/검증을 앞단에 두면 상당수 예방.
- **다중 창 분산 + GitHub 웹 업로드**: 작업본을 편집하는 동안 같은 파일을 웹에서 업로드("Add files via upload") → 실제 분기/병합 발생. **코드 파이프라인 하나로** 유지 권장.
- **요구의 사후 등장**: 중간 제거·유형별 기수·기본 시점 등은 중반에 나와 재작업을 유발(시점 탭을 3단계로 만들었다가 2단계로). 데이터 모델/스코프를 초반에 정의했으면 감소.

## 7. AI/에이전트가 잘한 점 (해석)
- 매 이슈를 **라이브 Firestore 데이터로 재현·진단**(추측 아님).
- 5기 문제에서 **위험한 대량 DB 변경 대신 자동보정** 선택(안전·되돌리기 쉬움).
- 정성/정량 누수를 **구조적 근본 수정**(시점별 판정)으로 처리하고 두 경로(`qualResponseRows`, `buildQualPrompt`)에 일관 적용.
- 변경마다 **브라우저 검증** 후 보고, Change뿐 아니라 Report도 점검.
- 원격 분기 시 **상위집합 검증 후** 병합, 임시 우회 설정(launch.json)은 **커밋에서 제외/되돌림**.
- 점수 매핑처럼 **모호하고 결과에 영향 주는 결정은 질문**.

## 8. AI/에이전트가 개선할 점 (해석)
- **UX 기본값을 사용자가 발견하게 둠**: 시점 탭을 추가하며 "기본 탭을 무엇으로?"를 충분히 고려 안 함 → 사전 업로드가 사후 탭에 가려 "안 보임" 신고를 받음. (예측 가능한 누락)
- **교차 페이지 정합 선제 점검 부족**: Change 정성을 고친 뒤 Report의 정성 경로가 같은지 먼저 확인 안 함 → 사용자가 차이를 발견.
- **수동 검증 의존 + 회귀 테스트 부재**: 중간 제거가 `stats[2]`를 건드릴 수 있었음(코드 정독으로 막았지만 테스트가 더 안전).
- **환경 마찰 반복**: 미리보기 서버가 샌드박스로 프로젝트를 못 읽어 매번 우회(Bash 서버) + 세션 끊김 + 캐시 버전 수동 bump. Bash-서빙 우회가 신뢰 경로임을 더 빨리 확정했어야.

- **파생 필드는 단일 진실원천에서 도출**(cohort는 세션에서). 드리프트 버그 한 부류를 통째로 없앰.
- **문맥별로 분류**(주관식/객관식을 설문+시점 단위로). 절대 시점 간 풀링 금지.
- **코드 파이프라인 하나**. GitHub 웹 업로드 같은 우회 입력 금지.
- **분석 UI 전에 데이터 모델/불변식 정의**.
- **신고된 시나리오 + 형제 화면(Change↔Report)을 함께 검증**.
- **환경 셋업(서빙/미리보기)은 처음에 1회 확정**, 캐시버스팅 자동화.

## 13. 최종 한 줄 평가
- 운영을 막던 데이터 무결성/정합 버그를 실데이터 기반으로 견고하게 근본수정했고 전부 검증·배포했지만, 상당수가 입력단 데이터 불일치에서 비롯됐고 요구가 중반에 나와 재작업과 환경 마찰이 있었다.

## 14. 점수
- 과정 효율 점수(1~5): **3** (반응적 수정·재작업·환경 마찰 多)
- 결과 품질 점수(1~5): **4** (근본수정·검증·클린 커밋)
- 다음에 같은 방식으로 반복할지: **부분** — 유지: 명확한 버그신고 + 실데이터 진단 + 즉시 검증 루프 / 변경: 데이터 모델 선정의 + 파이프라인 단일화 + 자동 검증
