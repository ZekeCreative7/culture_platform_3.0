## 1. 한 문장
이건 **소수의 조직문화 담당자(운영자 본인)** 가 **세션(팀빌딩·팀장·크로스펑셔널)을 통해 "변화에 유연한 조직문화"를 만드는 일** 을 운영하고, **그 개입이 실제로 효과가 있었는지를 (기수·팀 단위) 변화량으로 보여주기** 위해 쓰는 도구다.

---

## 2. 목적과 위상
- **메인 목적/기능:** 변화에 유연한 조직문화 만들기 / 세 종류 세션의 기획 ➡️ 타겟팅 ➡️ 운영 ➡️ 가시화
- **측정의 위상:** 메인 기능을 돕는 부품 (측정이 목적이 아님)
- **v1 범위:** 북극성(전체 구조) 중 한 조각만 만든다 ➡️ **Session Logger (세션 기록기)**

---

## 3. 사용자 및 권한
- **사용자:** 운영자 1인 콕핏(입력/조회) + 경영진(조회자). 팀장 및 타 팀은 조회 불가.
- **⚠️ v1 스펙:** 다중 사용자 및 권한 제어 시스템은 **만들지 않는다.** 운영자 1인 기준으로 시작.

---

## 4. 측정 아키텍처 (인덱스 산식 분리)
- **Pulse Survey (산식 밖):** 연 1회 전사 단위 측정. 문항 통제 불가. 기초체력 인디케이터 및 기획 맥락용으로만 활용.
- **세션 설문 (산식 안):** 세션별 사전·중간·사후 측정. 문항 통제 가능. 개입군의 **변화량(사후-사전 평균)**을 추출하는 핵심 지표.
- **원칙:** 두 서베이를 하나의 산식으로 섞지 않으며, 분석 단위는 개인을 배제한 **기수(cohort)/팀 단위**로 제한한다.

---

## 5. 설계 원칙 (헌법 조항)
1. 사용자는 운영자 1인 콕핏 + 경영진 보고 화면 중심. 다중권한 아님.
2. 경영진 지표는 활동 참여율이 아니라 **기수·팀 단위의 '사전 ➡️ 사후 변화량(사후 평균 − 사전 평균)'** 이며, **응답 인원수(N)를 항상 병기**한다. 개인 매칭을 하지 않으므로 통계적 유의성이 아닌 **방향·크기 중심의 서술적 지표**로 해석한다.
3. 세션은 '일반 객체 + 종류 태그' 구조로 설계하여 하드코딩을 방지한다.
4. 측정 틀(5점 척도, 모름=결측)은 공통이나, 문항/차원은 세션 타입마다 다르게 적용한다.
5. 향후 확장성(멀티 회사, 캘린더 연동 등)은 데이터 구조만 대비하고 v1 기능에는 넣지 않는다.
6. 모든 주장은 측정한 수치로만 추적하며, 재식별 위험을 막기 위해 **개인 식별자는 일절 금지**한다.
7. "모름" 응답은 낮은 점수가 아닌 **결측치(Null)**로 처리하여 연산에서 제외한다.

## 6. 시스템 아키텍처 및 폴더 구조 (운영자 주도 설계)
이 시스템은 UI(화면), 비즈니스 로직(가공), 데이터베이스(저장)가 철저히 분리된 구조로 빌드한다. AI는 이 파일 구조를 임의로 변경할 수 없다.

```text
session-logger/
├── plan.md          # 단일 진실 공급원 (SSOT)
├── app.py           # [프론트엔드] Streamlit 화면 레이아웃 및 콕핏 UI 담당
├── db.py            # [데이터베이스] SQLite 연결 및 데이터 조회/생성(CRUD) 로직 담당
├── parser.py        # [비즈니스 로직] 구글 폼 CSV 파일 검증 및 결측치(Null) 가공 담당
└── schema.sql       # [데이터베이스] DB 테이블 규격 및 스키마 정의서

```

### 6.1 컴포넌트별 역할 제한

* **app.py:** 입력 폼, 파일 업로드 버튼, 그래프 표출만 담당. 내부에 SQL 쿼리나 CSV 파싱 로직 직접 작성 금지.
* **parser.py:** CSV 내 식별자 유무 검증 및 '모름' 응답을 `Null`로 정제하여 반환하는 파이프라인 역할만 수행.
* **db.py:** `schema.sql` 기반 DB 초기화, 정제 데이터 저장, 기수별 변화량($\text{사후} - \text{사전}$) 연산 쿼리 수행.

## 7. v1 범위 및 데이터 모델

### 7.1 데이터 모델 (A안: 익명 개별 응답 행 단위 저장)

```text
Session (세션)
  - id (PK) | team (팀 라벨) | hq (본부) | type (태그: 팀장|팀빌딩|크로스펑셔널) | duration_weeks | cohort (기수) | dates (시작/종료일)

Response (응답) — Session에 속함, 익명 행 단위 저장 (개인식별자 일절 없음)
  - id (PK) | session_id (FK) | cohort | phase (사전|중간|사후)
  - quant (문항별 정량 점수 맵 - Key-Value 구조, 모름=null)
  - qual (문항별 주관식 텍스트 맵 - Key-Value 구조)

CohortStat (기수 집계) — 내부 연산 뷰(View)
  - 기수 × phase × 문항 평균 | 사전 N / 사후 N | 변화량 = 사후 평균 - 사전 평균

```

### 7.2 기능 범위 (한다 / 안 한다)

* ✅ **한다:** 세션 정보 등록/관리, 구글 폼 CSV 파일 업로드 및 자동 파싱/적재, 정량+정성 데이터 매핑 저장, 기수·팀별 사전 ➡️ 사후 변화량 자동 계산, 시점별 응답 인원수(N) 기록 및 리스트 조회.
* ❌ **안 한다:** 개인별 추적/매칭, 대시보드 인덱스 가중 결합, Pulse 연동 분석, 스케줄링/캘린더 연동, 권한 시스템.

## 8. 설문 재설계: 팀장 세션 사전 설문 확정

* **응답/척도:** 기수 선택 + 익명 안내문 노출 / 표준 5점 척도 (Integer, Nullable / 모름=Null)
* **정량 문항:** [심리안전] q1~q3 | [사일로해소] q4~q6 | [회복/긴장] q7 | [전반분위기] q8
* **정성 문항:** [사전] q9 (이 모임에 기대하는 점) ➡️ [사후] q1~q8 동일 적용 + q10 (좋았던 점), q11 (운영진에게)

### 8.1 적용 범위 및 phase 운영

* **v1 적용 범위:** 본 설문 설계는 **팀장 세션에 한정**한다. 팀빌딩·크로스펑셔널 세션은 데이터 구조상 등록은 가능하나, 문항 세트는 v1에서 만들지 않는다.
* **phase 운영:** 사전·중간·사후는 **각각 별도의 구글 폼**으로 만든다. phase 값은 폼에 묻지 않고, CSV 적재 시 **운영자가 업로드 화면에서 지정**한다(중복 입력·오입력 방지). '중간'은 선택이며 있을 때만 적재.
* **구글 폼 설정:** "이메일 주소 수집" **끄기**, "응답 횟수 1회로 제한" **끄기** ➡️ 로그인 없이 익명 응답. QR은 동일 폼 URL을 가리킨다.

---

## 9. 제약 및 리스크 관리

* N수가 작으므로(기수당 ~6명) 통계적 유의성 검증보다 방향성과 변화 크기(방향·서술) 중심으로 해석한다.
* 신상 파생 식별자 사용을 엄금하며, 기개입된 3팀은 사후 데이터만 존재하므로 향후 세션부터 0주차 사전 설문을 필수 적용한다.
* **소규모 셀 마스킹:** 익명 행 단위 저장이라도, 특정 셀의 응답자가 극소수(1~2명)이면 사실상 개인이 특정될 수 있다. 따라서 모든 집계는 **기수/팀 단위로만** 보고하고, **표시 인원 N이 3 미만인 셀은 화면·보고서에 노출하지 않는다(마스킹).**

## 10. 화면 흐름 (운영자 콕핏)

Streamlit 화면 4개. 운영자는 위에서 아래 순서로 사용한다.

1. **세션 관리** — 세션 등록/조회 (team, hq, type, cohort, dates, duration_weeks).
2. **CSV 업로드** — ⓐ 세션 선택 → ⓑ **phase 선택(사전/중간/사후)** → ⓒ CSV 드롭 → ⓓ parser 검증 결과 미리보기(식별자 경고 / 컬럼 매핑 / 정상 N) → ⓔ 적재.
3. **변화량 조회(운영자)** — 기수×문항 사전·사후 평균과 변화량(Δ), 사전 N / 사후 N, 정성 텍스트 리스트.
4. **경영진 보고(조회 전용)** — 방향·크기 중심 요약. **셀 N<3 마스킹**, 개인식별자 없음.

## 11. 데이터 계약 (구글 폼 CSV ↔ DB)

AI가 컬럼 매핑을 임의로 발명하지 못하도록, **폼 문항 제목 맨 앞에 기계가독 태그를 붙인다.** parser는 한글 문항 전문이 아니라 이 `[태그]`로만 컬럼을 인식한다.

**문항 제목 규칙(구글 폼 작성 시):** 모든 문항 제목은 `[태그] 실제 질문문` 형태로 작성한다.

| 태그 | 의미 | 타입 / 값 |
|---|---|---|
| `[기수]` | 기수 선택 | 정수 |
| `[q1]`~`[q3]` | 심리안전 | 5점(아래 매핑), 모름=Null |
| `[q4]`~`[q6]` | 사일로 해소 | 5점, 모름=Null |
| `[q7]` | 회복/긴장 | 5점, 모름=Null |
| `[q8]` | 전반 분위기 | 5점, 모름=Null |
| `[q9]` | (사전) 기대하는 점 | 텍스트 |
| `[q10]` | (사후) 좋았던 점 | 텍스트 |
| `[q11]` | (사후) 운영진에게 | 텍스트 |

**정량 5점 값 매핑:** 매우 그렇다=5 / 그렇다=4 / 보통=3 / 그렇지 않다=2 / 전혀 그렇지 않다=1 / **모름·해당없음=Null**

**CSV → DB 적재 규칙:**
* 구글 폼 CSV의 첫 컬럼 `타임스탬프`는 무시. 나머지 헤더에서 선행 `[태그]`만 추출해 필드를 식별한다.
* `session_id`·`phase`는 **업로드 화면 선택값**에서 채운다(CSV 아님).
* `cohort`는 `[기수]` 컬럼에서 행 단위로 채운다.
* 한 CSV 행 = `Response` 한 줄.

**검증 실패 조건(적재 거부):** 직접 식별자 컬럼(이름·사번·이메일·전화 등) 발견 / 필수 태그(`[기수]`, `[q1]`~`[q8]`) 누락. ➡️ 화면에 사유 표시 후 중단한다.

## 12. v1 완료 정의 (Acceptance Criteria)

아래가 끝까지 동작하면 v1 완료다. 그 외 기능은 만들지 않는다.

1. 운영자가 팀장 세션 1개를 등록한다.
2. 사전 CSV를 업로드 → 식별자 검증 통과 → 적재.
3. 사후 CSV를 업로드 → 적재.
4. **기수 단위 사전·사후 평균, 변화량(Δ), 사전 N / 사후 N**이 화면에 표시된다.
5. 경영진 화면에서 **셀 N<3은 마스킹**되고, 개인식별자가 어디에도 존재하지 않는다.
